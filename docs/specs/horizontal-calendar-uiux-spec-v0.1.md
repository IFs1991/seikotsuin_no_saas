# Tiramisu Horizontal Calendar UI/UX Design Specification v0.1

## 0. 文書情報

| 項目 | 内容 |
|---|---|
| 文書名 | Horizontal Calendar UI/UX Design Specification |
| 対象範囲 | 予約管理カレンダー / シフト入力カレンダー / ブロック表示 |
| 想定利用者 | 本部管理者、院管理者、受付、スタッフ |

## 1. 目的

Tiramisuの予約管理・シフト入力・ブロック管理において、現場スタッフが直感的に理解できる横向きタイムテーブルUIを提供する。

本仕様の目的は以下である。

- 日本の医療・治療院現場で見慣れた横向き時間割に近いUIにする
- 予約、シフト、休憩、外部予定、ブロックを同一設計思想で扱う
- スタッフ別・ベッド別・リソース別の稼働状況を一画面で把握できるようにする
- 予約事故、二重予約、シフト入力ミスを減らす
- 将来的なGoogleカレンダー連携、LINE予約、SALON BOARDブロック同期の土台にする

---

## 2. 基本方針

### 2.1 横向きタイムテーブルを採用する

TiramisuのカレンダーUIは、以下を基本とする。

```text
縦軸: スタッフ / ベッド / リソース / 外部スケジュール
横軸: 時刻
カード: 予約 / シフト / 休憩 / ブロック / 外部予定
```

Google Calendarのような縦方向の時間軸ではなく、日本の院内予約表・紙のシフト表に近い横方向の時間軸を採用する。

### 2.2 正本はTiramisu DBとする

カレンダーUIはGoogle Calendar風にしてよいが、予約・シフト・ブロックの正本はTiramisu DBとする。

```text
予約SSOT = Tiramisu reservations
シフトSSOT = Tiramisu shifts
ブロックSSOT = Tiramisu blocks
Google Calendar = 外部表示 / 補助同期
```

Google Calendar APIを予約台帳・シフト台帳の正本にしない。

### 2.3 操作より視認性を優先する

初期MVPでは、高度なドラッグ操作より以下を優先する。

- 見やすいこと
- 誤クリックしにくいこと
- 予約枠・休憩・ブロックの違いが明確なこと
- スタッフ別/ベッド別の空き状況が一目で分かること
- 詳細パネルで操作を確定できること

### 2.4 テナント境界と権限をUIにも反映する

予約、シフト、ブロック、通知はすべて clinic_id を基準に表示範囲を制御する。

原則:

- 親テナント管理者は許可された子テナントの予約状況を切り替えて確認できる
- 子テナントの clinic_admin は所属クリニックまたは許可された子テナントのみ確認できる
- therapist / staff は自分に関係する予約・シフトを中心に表示する
- 権限がない作成・編集・キャンセル・通知送信操作は非表示または disabled にする
- UIで非表示にしても、最終的な制御はAPI/RLS/サーバー側ガードで行う

### 2.5 日付・時刻はJST基準で扱う

院内業務の表示基準は日本時間とする。

原則:

- カレンダー表示日は JST の日付で扱う
- 保存時はDBの既存方針に合わせるが、UI表示ではJSTへ正規化する
- 日跨ぎ予約はMVPでは原則扱わない
- 営業時間外予約は警告または権限者のみ許可とする
- 休診日・臨時休診日は予約作成時に警告する

## 3. 対象画面

### 3.1 予約管理カレンダー

対象ルート候補:

```text
/reservations
/reservations/calendar
```

主な目的:

- 予約一覧表示
- 予約作成
- 予約編集
- 予約キャンセル
- スタッフ別予約確認
- ベッド/リソース別予約確認
- ブロック確認

### 3.2 シフト入力カレンダー

対象ルート候補:

```text
/staff/shifts
/staff/schedule
```

主な目的:

- スタッフ勤務時間の入力
- 休憩時間の入力
- 休み入力
- 外部予定入力
- 繰り返しシフト設定
- Google Calendar連携状態の確認

### 3.3 ブロック管理

対象ルート候補:

```text
/blocks
/reservations/blocks
```

主な目的:

- 予約不可枠の作成
- スタッフ不在枠の作成
- ベッド/リソース使用不可枠の作成
- 会議/研修/院内イベントの登録

## 4. 画面レイアウト

## 4.1 共通レイアウト

```text
┌────────────────────────────────────────────────────────────┐
│ Header: title / date navigation / view switch / actions     │
├──────────────┬───────────────────────────────┬─────────────┤
│ Sidebar      │ Horizontal Timetable           │ Detail Panel │
│ Navigation   │                               │             │
└──────────────┴───────────────────────────────┴─────────────┘
```

### Header

表示要素:

- 画面タイトル
- 今日ボタン
- 前日/翌日 または 前週/翌週ボタン
- 日付/期間表示
- 表示切替
  - 日
  - 週
  - 月
- 表示対象切替
  - スタッフ
  - ベッド
  - スタッフ + ベッド
- 検索
- フィルタ
- 作成ボタン

### Main Timetable

表示要素:

- 横軸: 時刻
- 縦軸: スタッフ/リソース
- カード: 予約/勤務/休憩/ブロック
- 現在時刻ライン
- 営業時間外の薄い背景
- 休診日/休日の背景

### Detail Panel

右側に選択中の予約・シフト・ブロックの詳細を表示する。

原則:

- 一覧上で直接すべて編集させない
- カード選択 → 詳細パネル → 編集/保存
- 破壊的操作は確認を挟む

# 5. 予約管理カレンダー仕様

## 5.1 表示構造

### 基本構造

```text
横軸: 09:00 - 21:00
縦軸: スタッフ / ベッド / リソース
カード: 予約 / ブロック / 休憩 / 外部予定
```

### セクション

予約管理画面では、以下のセクションを表示できる。

1. スタッフ
2. ベッド/施術室
3. 外部予定
4. ブロック

MVPでは以下を推奨する。

```text
上段: スタッフ別
下段: ベッド/リソース別
```

## 5.2 予約カード

### 表示内容

予約カードには以下を表示する。

| 項目 | 表示 |
|---|---|
| 時刻 | 必須 |
| 患者名 | 必須 |
| メニュー | 必須 |
| 担当者 | 任意 |
| ステータス | 色/バッジ |
| 初回/再来 | 任意 |
| チャネル | 任意 |

### 表示例

```text
09:00 - 09:50
山本 一郎 様
骨盤矯正
山田 太郎
```

## 5.3 色分け

| 種別 | 色方針 |
|---|---|
| 初診/カウンセリング | 淡い青 |
| 再診 | 淡い緑 |
| 施術/治療 | 淡い橙 |
| 検査/処置 | 淡い紫 |
| 点滴/注射 | 淡いピンク |
| 休憩 | グレー |
| 外部予定 | 斜線 + グレー/紫 |
| ブロック | 斜線 + グレー |
| キャンセル | 低彩度 + 取り消し線または非表示切替 |

### 注意

色だけに依存しない。必ずラベルまたはアイコンも併用する。

## 5.4 操作

### カードクリック

予約カードをクリックすると右側に予約詳細を表示する。

右側パネルの内容:

- ステータス
- 患者名
- 電話番号
- 日付
- 時刻
- メニュー
- 担当者
- ベッド/リソース
- メモ
- 内部メモ
- 編集ボタン
- キャンセルボタン

### 空き枠クリック

空き枠クリック時の挙動:

```text
空き枠クリック
↓
予約作成パネルを開く
↓
クリックした時刻・スタッフ・リソースを初期値に入れる
```

MVPではクリック作成でよい。ドラッグ作成は後回しでもよい。

### ドラッグ操作

将来的には以下をサポートする。

- 予約時間変更
- 担当者変更
- ベッド変更

ただし、MVPでは必須ではない。

ドラッグ変更時は必ず以下を行う。

- 衝突チェック
- 確認モーダル
- 通知送信有無の確認
- 監査ログ記録

## 5.5 衝突表示

予約衝突がある場合は以下を表示する。

- カード枠を赤にする
- 警告アイコン表示
- 詳細パネルに衝突理由表示
- 保存不可

衝突例:

- 同一スタッフの同時間帯予約
- 同一ベッドの同時間帯予約
- ブロック時間帯への予約
- シフト外予約
- 休憩時間への予約

## 5.6 予約ステータスと通知状態

予約カードでは、業務上重要な状態を視認できるようにする。

### 予約ステータス

| ステータス | 意味 | 表示方針 |
|---|---|---|
| unconfirmed | 未確定 | 注意色 + 未確定ラベル |
| tentative | 仮予約 | 点線枠または薄い色 |
| confirmed | 確定 | 通常表示 |
| arrived | 来院済み | チェックアイコン |
| completed | 完了 | 低彩度または完了ラベル |
| cancelled | キャンセル | 低彩度 + 取消ラベル |
| no_show | 無断キャンセル | 警告色 + no-showラベル |

### 通知状態

| 通知状態 | 意味 | 表示方針 |
|---|---|---|
| not_required | 通知不要 | 表示しない |
| pending | 送信待ち | 時計アイコン |
| sent | 送信済み | チェックアイコン |
| failed | 送信失敗 | 警告アイコン |
| skipped | 手動で送信しない | ミュートアイコン |

予約カードには通知本文を直接表示しない。詳細パネルに通知履歴と再送導線を表示する。

## 5.7 予約作成・変更時の確認

予約作成・変更・キャンセルでは、保存前に以下を確認する。

- 同一スタッフの予約衝突
- 同一ベッド/リソースの予約衝突
- ブロック/休憩/シフト外との衝突
- 予約日時変更に伴う通知送信有無
- キャンセル通知送信有無
- 患者へ見えるメモと内部メモの混在

MVPでは、詳細パネルまたは確認モーダルで「通知する / 通知しない」を選べればよい。

# 6. シフト入力カレンダー仕様

## 6.1 表示構造

```text
横軸: 08:00 - 21:00
縦軸: スタッフ
カード: 勤務 / 休憩 / 休み / 外部予定 / 会議 / 研修
```

### 推奨表示

- 日表示: 1日のスタッフ別シフト
- 週表示: スタッフごとの週シフト
- 月表示: 詳細ではなく集計中心

MVPでは **日表示 + 週表示** を優先する。

## 6.2 シフトカード

### 表示内容

| 項目 | 表示 |
|---|---|
| 時刻 | 必須 |
| 区分 | 必須 |
| メモ | 任意 |
| Google連携状態 | 任意 |
| 確定/下書き | 任意 |

### 表示例

```text
09:00 - 18:00
勤務
```

```text
13:00 - 14:00
休憩
```

## 6.3 区分

| 区分 | 意味 |
|---|---|
| work | 勤務 |
| break | 休憩 |
| day_off | 休み |
| external | 外部予定 |
| meeting | 会議 |
| training | 研修 |
| unavailable | 予約不可 |

## 6.4 色分け

| 種別 | 色方針 |
|---|---|
| 勤務 | 淡い緑 |
| 休憩 | 淡い橙 |
| 休み | 淡い赤/ピンク |
| 外部予定 | 淡い紫 + 斜線 |
| 会議/研修 | グレー/紫 |
| 未確定 | 点線枠 |
| 確定 | 通常枠 |

## 6.5 操作

### 空き枠ドラッグ

シフト入力では、予約管理よりもドラッグ操作を優先する。

```text
スタッフ行の空き時間をドラッグ
↓
シフト作成パネルを開く
↓
開始/終了時刻を自動入力
```

### カードクリック

シフトカードクリックで右側詳細パネルを開く。

表示項目:

- ステータス
- スタッフ
- 日付
- シフト時間
- 休憩時間
- 区分
- メモ
- 繰り返し
- Googleカレンダー連携状態
- 削除ボタン

### 繰り返し設定

MVPでは簡易対応。

- 繰り返さない
- 毎週同じ曜日
- 平日のみ
- カスタム

初期はUIだけ用意し、実装は段階的でよい。

# 7. ブロック表示仕様

## 7.1 ブロックの目的

ブロックは予約不可枠を表す。

例:

- 休憩
- 外部予定
- 会議
- 研修
- 院内イベント
- 設備メンテナンス
- ベッド使用不可
- スタッフ不在

## 7.2 表示場所

ブロックは以下に表示する。

- 予約管理カレンダー
- シフト入力カレンダー
- Blocks画面

## 7.3 表示ルール

- 斜線背景を使用する
- 通常予約と明確に区別する
- 理由を表示する
- 操作権限がない場合は閲覧のみ

## 7.4 SALON BOARD同期との関係

将来的にSALON BOARDへ同期する場合、最初に同期する対象は予約ではなくブロックとする。

```text
Tiramisu block
↓
Sync queue
↓
SALON BOARD RPA / Adapter
↓
同期結果をTiramisuへ返す
```

# 8. Google Calendar連携UI

## 8.1 基本方針

Google Calendarは外部同期先/同期元として扱う。

画面上では以下を表示する。

- 連携状態
- 連携カレンダー名
- 最終同期日時
- 同期方向
- 連携設定ボタン

## 8.2 表示例

```text
Googleカレンダー連携
● 連携中（メインカレンダー）
[連携設定を開く]
```

## 8.3 同期方向

| 同期方向 | 意味 | 初期推奨 |
|---|---|---|
| export_only | Tiramisu → Google Calendar | 推奨 |
| import_review | Google Calendar → Tiramisu候補 | 次段階 |
| two_way | 双方向同期 | 後回し |

MVPでは `export_only` を推奨する。

# 9. 通知機能

## 9.1 基本方針

通知は予約・シフト・ブロック操作に付随する補助機能として扱う。予約データそのものの正本にはしない。

```text
reservation / shift / block
↓
notification intent
↓
notification queue / outbox
↓
email / LINE / SMS / in-app
↓
notification history
```

MVPでは、まず院内管理者・スタッフ向けの画面内通知とメール通知の状態表示を優先する。LINE/SMSは将来拡張とする。

## 9.2 通知対象イベント

予約管理では以下のイベントを通知対象にする。

| イベント | 患者向け通知 | スタッフ向け通知 | MVP |
|---|---:|---:|---:|
| 予約作成 | あり | あり | 必須 |
| 予約日時変更 | あり | あり | 必須 |
| 担当スタッフ変更 | 任意 | あり | 必須 |
| ベッド/リソース変更 | なし | 任意 | 任意 |
| 予約キャンセル | あり | あり | 必須 |
| 未確定予約の承認 | あり | あり | 必須 |
| no-show登録 | なし | あり | 任意 |
| ブロック作成/変更 | なし | 関係スタッフのみ | 任意 |
| シフト変更 | なし | 対象スタッフ | 次段階 |

## 9.3 通知チャネル

| チャネル | 用途 | MVP |
|---|---|---|
| in_app | 管理画面内の未読通知・アラート | 必須 |
| email | 予約作成/変更/キャンセル通知 | 必須 |
| LINE | 患者向け予約通知・リマインド | 後回し |
| SMS | 電話番号のみ患者への通知 | 後回し |
| webhook | 外部連携・自動化 | 後回し |

## 9.4 通知UI

### ヘッダー通知

ヘッダーには以下を表示する。

- 未確定予約件数
- 未読通知件数
- 通知送信失敗件数
- クリック時の通知一覧モーダルまたは通知ページ導線

### 予約詳細パネル

予約詳細パネルには以下を表示する。

- 最終通知日時
- 通知チャネル
- 送信状態
- 送信失敗理由
- 再送ボタン
- 通知せず保存するチェック/トグル

### 予約作成・編集フォーム

予約作成・編集フォームには以下を表示する。

- 患者へ通知する
- 担当スタッフへ通知する
- 変更理由
- 患者向けメッセージプレビュー

MVPではメッセージ本文の完全な自由編集は不要。テンプレート文 + 変更理由の入力でよい。

## 9.5 リマインド通知

予約リマインドは将来機能として扱う。

候補:

- 前日リマインド
- 当日朝リマインド
- 予約2時間前リマインド
- 無断キャンセル防止通知

MVPでは予約作成・変更・キャンセルのトランザクション通知を優先し、定期リマインドは後回しにする。

## 9.6 通知失敗時の扱い

通知送信に失敗しても、予約保存そのものを必ず失敗させるとは限らない。

原則:

- 予約保存と通知送信は分離する
- 通知送信は outbox / queue 経由にする
- 送信失敗は予約詳細パネルと通知一覧に表示する
- 管理者または権限者が再送できる
- 同じ通知の二重送信を防止する

## 9.7 通知テンプレート

通知テンプレートは院ごとのカスタマイズを想定する。

MVPで必要なテンプレート:

- 予約作成
- 予約変更
- 予約キャンセル
- 未確定予約の承認
- スタッフ向け担当予約通知

将来的には、親テナントの共通テンプレートを子テナントが上書きできる設計にする。

## 9.8 通知の監査ログ

通知に関する操作は監査ログに残す。

記録対象:

- 誰が通知を発火したか
- 対象予約ID
- 対象患者/スタッフ
- 通知チャネル
- 送信状態
- 送信日時
- 失敗理由
- 再送回数

# 10. ビュー切替

## 10.1 日表示

目的:

- 当日の稼働確認
- 受付/院長の実運用
- 予約作成
- シフト入力

横軸:

```text
08:00 - 21:00
```

縦軸:

```text
スタッフ / ベッド / リソース
```

## 10.2 週表示

目的:

- 週単位の予約密度確認
- シフト計画
- スタッフ稼働比較

予約管理での週表示は、情報過多になりやすいため以下を検討する。

- 日付をタブで切り替える
- 1日単位横タイムテーブルを維持する
- 週表示では集計/密度表示に寄せる

シフト入力では週表示を優先してよい。

# 16. 現場オペレーション機能とDB影響

## 16.1 追加検討すべき現場機能

横向きカレンダーを単なる予約表示ではなく、受付・施術・会計前後の業務ツールとして使うため、以下の機能を段階的に検討する。

| 機能 | 目的 | DB影響 |
|---|---|---|
| 表示密度切替 | 標準/コンパクト/詳細で予約表の見え方を変える | なし |
| 当日業務ステータス | 来院済み、施術中、会計待ちなどを可視化する | あり |
| 受付向け次アクション | 来院、施術開始、完了、会計へ進む、次回予約を作る | 一部あり |
| 予約経路表示 | 店頭/電話/Web/LINE/Google/SALON BOARDを区別する | あり |
| 初診/再診/保険/自費/回数券バッジ | 受付判断と施術準備を早くする | あり |
| キャンセル待ち | 空き枠発生時に候補患者へ連絡する | あり |
| 設定不足警告 | スタッフ/メニュー/営業時間/シフト未設定を知らせる | なしから一部あり |
| 印刷/日次確認 | 紙運用・朝礼・終礼で当日予約表を確認する | なし |
| 変更履歴表示 | 誰がいつ予約を変更したか確認する | 既存DB確認またはあり |
| 権限別操作マトリクス | roleごとに表示/作成/編集/キャンセル/通知を制御する | なしから一部あり |

## 16.2 DB影響なしで先に進める範囲

以下は既存の予約・メニュー・リソース・権限データを使って、UI/UX改善として先行できる。

- 予約表の表示密度切替
- 予約カードの見た目改善
- 予約/ブロック/キャンセルの色・ラベル・アイコン整理
- 右詳細パネルの情報設計改善
- 既存ステータスを使った表示強化
- 空き枠クリックからの予約作成導線改善
- 既存roleによる操作ボタンの出し分け
- スタッフ/メニュー/リソース未登録など、既存データから判定できる設定不足警告
- 本日の予約表の印刷向けレイアウト

この範囲では Supabase migration は不要とする。

## 16.3 既存DBで吸収できるか確認する範囲

以下は既存テーブル・既存カラム・既存ログで表現できる可能性があるため、実装前にDB確認を行う。

- 予約作成者/更新者/キャンセル実行者の表示
- 予約変更履歴の表示
- 通知送信状態の表示
- 通知失敗理由の表示
- 予約経路が既存カラムまたはメモで代替できるか
- 患者種別や初診/再診を既存 customer / reservation データから判定できるか

既存DBで表現できない場合は、仕様書とロールバック方針を用意して migration を検討する。

## 16.4 DB変更が必要になりやすい範囲

以下を本格実装する場合は、DB設計・RLS・API・監査ログを含めた変更が必要になる可能性が高い。

- `in_treatment` など施術中ステータスの追加
- `waiting_payment` など会計待ちステータスの追加
- 予約経路を正規化して保存するカラムまたはテーブル
- キャンセル待ちリスト
- 通知outbox / queue
- 通知履歴、再送回数、失敗理由
- 通知テンプレートの親テナント/子テナント継承
- 回数券、保険/自費、会計との接続
- 予約変更履歴を永続化する監査テーブル

DB変更を伴う場合は、以下を必須とする。

- 変更対象テーブルとカラムの明記
- RLS/tenant boundary の影響確認
- APIガードの影響確認
- 既存データ移行方針
- ロールバックSQLまたはロールバック手順
- `docs/stabilization/DoD-v0.1.md` の該当項目への影響確認

# 19. 非推奨

以下は避ける。

## 19.1 Google Calendarを正本にする

避ける理由:

- 業務データが不足する
- clinic_id / role / RLS と合わない
- 予約衝突判定が不安定になる
- 売上/日報/患者管理と接続しにくい

## 19.2 縦型タイムテーブルのみで固定する

避ける理由:

- 日本の院内運用では横向き予約表の方が直感的なケースが多い
- スタッフ/ベッド比較がしづらい
- 紙の運用からの移行負荷が上がる

## 19.3 カードに情報を詰め込みすぎる

避ける理由:

- 視認性が落ちる
- 誤クリックが増える
- 詳細パネルの意味が薄れる

## 19.4 ドラッグ操作をMVP必須にする

避ける理由:

- 実装難度が上がる
- 衝突/通知/監査ログが複雑化する
- 初期品質が落ちやすい

# 20. 結論

TiramisuのカレンダーUIは、Google Calendarそのものを模倣するのではなく、日本の治療院・整骨院業務に合う横向きタイムテーブルとして設計する。

基本方針は以下で固定する。

```text
横軸 = 時刻
縦軸 = スタッフ / ベッド / リソース
カード = 予約 / シフト / 休憩 / ブロック / 外部予定
詳細操作 = 右側パネル
正本 = Tiramisu DB
Google Calendar = 外部同期/表示補助
```

この設計により、現場の紙予約表・紙シフト表に近い認知負荷で、Tiramisuの予約、シフト、ブロック、外部連携を統合できる。
