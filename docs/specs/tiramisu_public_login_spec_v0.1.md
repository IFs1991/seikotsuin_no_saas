# Tiramisu 公開導線・ログイン導線再設計仕様書 v0.2

## 要約

現状の Tiramisu は認証ロジック自体は存在するが、公開ページと認証後ページの UI 境界が弱い。  
その結果、未認証ユーザーから見ると「いきなり SaaS 本体が出てくる」印象になりやすく、入口としての理解コストが高い。

本仕様書の目的は、以下の 5 点を実現すること。

1. 公開面と認証後面をレイアウトごとに分離する  
2. `/` を入口ページとして再定義する  
3. スタッフ / 管理者のログイン導線を明確化する  
4. 境界仕様だけはテストで固定する  
5. AI に実装させる際の役割分担を明確にする  

---

## 背景

現状の構成では、`src/app/layout.tsx` で全ページが `ClientLayout` に包まれている。  
`ClientLayout` には Header / Sidebar / MobileBottomNav / profile 読み込みが常設されているため、公開ページやログインページも「認証後アプリの殻」を被る構造になっている。

さらに `/` は公開ホームではあるが、内容が以下の中間状態になっている。

- LP ではない
- ログインページでもない
- 完全なアプリホームでもない

そのため、初見ユーザーにとって以下が曖昧になる。

- 何のサービスか
- 自分はどこから入るべきか
- まだ公開ページなのか、もうアプリ内部なのか

---

## 現状認識

### 事実

- `/login` と `/admin/login` は既に存在する
- middleware により `/dashboard` `/reservations` `/patients` `/admin` 等は保護されている
- 未認証時は staff 系は `/login`、admin 系は `/admin/login` へリダイレクトされる
- ただし UI レイヤーでは公開面と認証後面の境界が弱い
- テスト基盤として Jest / Playwright は既に存在するため、今回の改修でゼロから導入する必要はない

### 解釈

問題は「ログインページが無い」ことではない。  
問題は **入口設計とレイアウト設計が未分離** なことにある。

---

## 目的

### 主目的

- 未認証ユーザーにとって自然な入口を作る
- スタッフと管理者の導線を迷わせない
- 認証後のアプリ画面を「中に入った後の体験」として明確に分離する
- CS / 問い合わせの発生源を減らす

### 副次目的

- 今後の公開予約フォームや LP 追加に耐えやすい構造へ寄せる
- 認証済みユーザーの初回到達画面を制御しやすくする
- UI/UX 改修の責務境界を明確にする
- AI による実装時の暴走を防ぐ

---

## スコープ

本仕様の対象は以下。

- 公開トップ `/`
- スタッフログイン `/login`
- 管理者ログイン `/admin/login`
- ルートグループ構成
- root layout / app layout / public layout の責務整理
- 認証済み時の `/` 挙動
- CTA / 文言 / ナビゲーション整理
- この改修に必要な最小テスト
- AI 実装時の役割分担

本仕様の対象外は以下。

- 決済導線の詳細実装
- 公開予約フォーム本体の実装
- 管理画面・予約画面内部の詳細 UI 改修
- 認証方式そのものの刷新
- Supabase Auth の基盤変更
- LP の本格的な営業最適化
- ビジュアル全面刷新

## 基本方針

### 設計方針

- 公開面と認証後面は **レイアウトごと分離** する
- ログインはページ単体でなく **入口導線全体** として設計する
- `/` は未認証向け入口ページとする
- 認証済みユーザーに対しては `/` を滞在ページにしない
- 認証後の Header / Sidebar は app 領域に閉じ込める

### UX 方針

- 「どこから入るか」を迷わせない
- 「何のサービスか」を 3 秒以内に理解できるようにする
- CTA は増やしすぎない
- ダッシュボードや AI チャットなど、入口として弱い導線は前面に出さない
- 本体感のある UI はログイン後に初めて出す

### 実装方針

- 大規模改修にしない
- 認証ロジックを不用意に触らない
- まず境界を分ける
- 境界はテストで固定する
- 見た目より導線優先で進める

## ルート構成方針

### 目標構成

```txt
src/app/
  layout.tsx

  (public)/
    layout.tsx
    page.tsx
    login/page.tsx
    admin/login/page.tsx
    invite/page.tsx
    privacy/page.tsx
    terms/page.tsx

  (app)/
    layout.tsx
    dashboard/page.tsx
    reservations/page.tsx
    patients/page.tsx
    revenue/page.tsx
    staff/page.tsx
    admin/...
    onboarding/page.tsx
    chat/page.tsx
```

### 役割

#### `src/app/layout.tsx`
- 最小の root layout のみ
- html / body / metadata のみ
- Header / Sidebar / profile fetch は置かない

#### `src/app/(public)/layout.tsx`
- 公開ページ専用レイアウト
- シンプルなヘッダーまたはヘッダー無し
- Sidebar / MobileBottomNav なし
- 認証依存の provider を極力持たない

#### `src/app/(app)/layout.tsx`
- 認証後アプリ専用レイアウト
- 現在の `ClientLayout` 相当をここへ移す
- Header / Sidebar / MobileBottomNav / profile / clinic 選択をここに閉じ込める

- 「一般スタッフは左側メニューから〜」のような内部前提説明
- システム状態表示

- サブコピー: 予約・患者・運営管理を一元化し、現場と管理をつなぐ

### 目的

### 要件
- 認証が通っている画面のみで app shell を表示
- profile / clinic / notification 取得はここでのみ実施

## リダイレクト仕様

## 1. 未認証ユーザー

### ルール
- `/dashboard` `/reservations` `/patients` `/staff` 等へ直接アクセス
  - `/login?redirectTo=...` へ遷移
- `/admin/**` へ直接アクセス
  - `/admin/login?redirectTo=...` へ遷移

### 備考
この挙動は基本的に既存 middleware の考え方を維持する。

---

## 2. 認証済みユーザーが `/` へアクセスした場合

### ルール
- staff / clinic side user
  - `/reservations` または `/dashboard` へ遷移
- admin user
  - `/admin` または `/dashboard` へ遷移

### 推奨
運用上の主画面が予約なら `/reservations` を優先する。  
現場オペレーション主体ならその方が自然。

---

## 3. 認証済みユーザーが `/login` `/admin/login` へアクセスした場合

### ルール
- 既に認証済みなら役割に応じたホームへ返す
- ログインページに滞在させない

## 情報設計ルール

### 入口ページで見せるべきもの
- 何のサービスか
- 誰向けか
- 何を改善するか
- どこから入るか

### 入口ページで見せないもの
- アプリ内部の各機能導線
- 内部メニュー前提の説明
- 権限前提の文脈
- 開発者視点のシステム状態表示
- 未認証ユーザーには意味の薄いダッシュボードリンク

## テスト方針

## 結論

全面 TDD は不要。  
ただし、**公開面 / 認証後面 / 認証遷移の境界だけは test-first で固定する。**

### 理由
今回の本丸はビジュアルではなく、以下の仕様契約だから。

- どの URL が公開か
- どの URL が認証必須か
- 未認証時にどこへ飛ぶか
- 認証済み時に `/` をどう扱うか
- どの画面に app shell が出るか

この部分は壊れると UX が崩れ、問い合わせも増える。  
一方で文言や余白まで固定すると開発速度を落とす。

---

## テスト対象

### A. 境界テスト（必須）
- 未認証 + `/reservations` → `/login?redirectTo=/reservations`
- 未認証 + `/admin` → `/admin/login?redirectTo=/admin`
- 認証済み clinic user + `/` → `/reservations` or `/dashboard`
- 認証済み admin + `/` → `/admin` or `/dashboard`

### B. レイアウト分離テスト（必須）
- `/login` に Sidebar が表示されない
- `/admin/login` に Sidebar が表示されない
- `/dashboard` に app shell が表示される
- `/reservations` に app shell が表示される

### C. スモーク E2E（推奨）
- 公開トップからスタッフログインへ遷移できる
- 公開トップから管理者ログインへ遷移できる
- ログイン後に期待通りのホームへ到達する

---

## テスト対象外

以下は今回固定しない。

- コピー文言の細部
- 余白や配色の微差
- LP 的な見せ方の細かい変更
- CTA 順序の軽微な調整
- ビジュアルデザインの微差分

---

## AI 実装方針

## 結論

AI に「全部やって」と投げるのは禁止。  
今回の改修は役割を分けた方が圧倒的に安定する。

### 採用する役割
1. UX アーキテクト
2. テスト設計者
3. リファクタ実装者
4. レビュアー

---

## AI 役割定義

## 1. UX アーキテクト

### 目的
- 公開面 / 認証後面 / ログイン導線の責務を定義する
- route group 構成を決める
- 画面遷移を整理する

### 入力
- 本仕様書
- 現在のルーティング構成
- 既存レイアウト構造

### 出力
- 画面一覧
- ルーティング表
- 遷移表
- layout 責務表
- public / app 分離方針

### 禁止
- いきなりコードを書き始めること
- 認証ロジックの再設計に踏み込むこと
- デザインシステム刷新の提案

---

## 2. テスト設計者

### 目的
- 壊れると致命的な境界仕様を抽出する
- Jest / Playwright で何を固定するか整理する

### 入力
- UX アーキテクトが定義した遷移仕様
- middleware の現状仕様
- 既存テスト基盤

### 出力
- テストケース一覧
- middleware contract test の設計
- layout separation test の設計
- smoke e2e test の設計

### 禁止
- 不必要なスナップショットテスト増殖
- 文言単位の過剰固定
- 微細UI差分までテスト対象にすること

---

## 3. リファクタ実装者

### 目的
- 仕様に従って最小差分で実装する

### 実装対象
- `(public)` `(app)` route group 導入
- `ClientLayout` の app 側移設
- `/` の再構成
- `/login` `/admin/login` の public 化
- 認証済みユーザーの `/` ハンドリング

### 出力
- 実コード
- 必要最小限の差分
- build / lint / test 通過可能状態

### 制約
- 認証方式を変更しない
- middleware の責務を増やしすぎない
- 関係ない UI をついでに触らない
- ダッシュボードや予約画面の内部 UX 改修を始めない

---

## 4. レビュアー

### 目的
- 過剰改修を止める
- 仕様逸脱を検出する
- public / app 境界漏れを検出する

### 入力
- 実装差分
- 仕様書
- テスト結果

### 出力
- 仕様逸脱指摘
- 過剰実装の削除提案
- リグレッション懸念
- 追加テストが必要な境界の指摘

### 禁止
- 実装を全面書き直しさせること
- 未来の抽象化を増やすこと
- 別件の改善要求を混ぜること

---

## AI への依頼順

### 正しい順番
1. UX アーキテクト
2. テスト設計者
3. リファクタ実装者
4. レビュアー

### 間違った順番
- いきなり実装者に全部投げる
- 仕様未確定のままコードから入る
- レビュー時に別件改善を混ぜる

## 非機能要件

### パフォーマンス
- 公開ページでは不要な認証依存 fetch を避ける
- 公開レイアウトで profile / clinic 情報を取得しない
- 初回表示速度を改善する

### 保守性
- 公開面と認証後面の責務をコード構造で分離する
- 今後の LP 追加、予約公開フォーム追加に耐えられる構造にする
- 仕様境界をテストで固定し、見た目は柔軟にする

### セキュリティ
- 認証保護の責務は引き続き middleware が持つ
- UI 分離は UX 改善であり、認可の代替ではない
- 公開ページに内部導線を出しすぎない

### 開発運用
- AI に役割を分けて依頼する
- 仕様未確定のままコード生成を走らせない
- 不要なついで改修を防ぐ

## 受け入れ基準

### 必須
- `/` が未認証ユーザー向け入口ページとして成立している
- `/login` と `/admin/login` が app shell なしで表示される
- `/dashboard` `/reservations` `/admin` など認証後ページのみ app shell を表示する
- 認証済みユーザーは `/` に滞在せず適切なホームへ遷移する
- 未認証ユーザーが保護ページへ入ろうとすると適切なログインページへ飛ぶ
- 境界仕様に関する最低限の Jest / Playwright テストが存在する

### 望ましい
- ログイン前後の世界観が視覚的に明確に変わる
- CS で「どこから入るの？」という問い合わせが減る
- 今後の公開予約フォーム追加時に layout を増やさず対応できる
- AI 実装時に不要改修が減る

## 非目標

以下はこの仕様ではやらない。

- LP を本格的な営業ページとして作り込む
- 決済導線の自動化
- 管理者 / スタッフ認証方式の大幅刷新
- 認可ロジックの全面改修
- 予約画面内部の詳細な導線最適化
- ビジュアル刷新プロジェクト化
- 全面 TDD 化

---

## 実装メモ

### 現在の本丸
- root layout で全ページが `ClientLayout` に包まれていること
- 公開ページにも Header / Sidebar が出ること
- `/` が入口ページとして曖昧なこと
- AI に一括実装させると関係ない箇所まで触りやすいこと

### 優先順位
1. layout 分離
2. `/` 再定義
3. ログイン導線整理
4. 認証済みリダイレクト最適化
5. 境界テスト追加
6. AI 役割分担の徹底

---

## 判断

この改修は見た目の改善ではない。  
**入口の意味づけ、責務分離、境界固定の改修** である。

今のままでも機能は使えるが、SaaS としての第一印象と導線品質は弱い。  
特に今後、公開予約フォームや導入導線を足す前提なら、ここを先に直す方がペイする。

また、今回の改修は AI と相性が良い一方、役割を分けないと過剰改修しやすい。  
したがって、仕様・テスト・実装・レビューを分離して進める。

---

## 結論

必要なのは「ログインページを追加すること」ではなく、  
**公開面と認証後面を分離し、`/` を入口ページとして再定義し、その境界仕様だけをテストで固定すること** である。

最優先でやるべきことは以下の 5 つ。

1. `(public)` と `(app)` で route group を分ける  
2. `ClientLayout` を `(app)` 側へ閉じ込める
