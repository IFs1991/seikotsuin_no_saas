## PR-01 recovery 用スキーマ / 型追加

### 目的

- forgot-password と reset-password が依存する最小のスキーマと型を追加する。

### 背景

[auth.ts](/C:/Users/seekf/Desktop/seikotsuin_management_saas/src/lib/schemas/auth.ts) には既に

- `passwordResetSchema`
- `passwordChangeSchema`

があるが、recovery 用の「新パスワード + 確認」は存在しない。

また、`passwordChangeSchema` は `currentPassword` を必須とするため recovery フローには不適切。

### 変更対象

- `src/lib/schemas/auth.ts`

### 実装ポイント

- `passwordRecoverySchema` を追加する
- `password`: `passwordSchema`
- `confirmPassword`: string
- `password === confirmPassword` を検証する
- 必要なら型を追加する
  - `PasswordRecoveryFormData`

### 完了条件

- recovery 用の新スキーマが追加される
- `passwordChangeSchema` を流用しないことがコード上でも明確になる

### 検証

- unit: 一致 / 不一致 / 弱いパスワードの検証

## PR-03 `/forgot-password/actions.ts` 実装

### 目的

- recovery メール送信の Server Action を実装する。

### 変更対象

- `src/app/(public)/forgot-password/actions.ts`

### 実装ポイント

#### 1. `resetPasswordForEmail(...)` を利用する

以下の契約で呼ぶ。

```ts
supabase.auth.resetPasswordForEmail(email, {
  redirectTo: `${appUrl}/admin/callback?next=/reset-password/${source}`,
});
```

#### 2. `source` は `admin | clinic` に正規化する

- 未指定や不正値は `clinic` に倒す
- query をそのまま trust しない

#### 3. 非列挙型レスポンス

- 存在しないメールでも同一の成功文言を返す
- provider の生メッセージをそのまま返さない

#### 4. 監査ログ

最低限、以下を記録する。

- `password_reset_requested`

#### 5. `NEXT_PUBLIC_APP_URL` は `assertEnv` で取得する

- フォールバックを持たない

### 完了条件

- `resetPasswordForEmail()` を使って recovery メール送信を開始できる
- 存在有無を開示しない
- `source` に応じて `redirectTo` が切り替わる

### 検証

- unit/integration:
  - 正常系
  - 不正 source
  - Supabase エラー時も同一文言

## PR-04 `/reset-password/[source]` UI / action 実装

### 目的

- recovery メールから戻ったユーザーが新しいパスワードを設定できるようにする。

### 変更対象

- `src/app/(public)/reset-password/[source]/page.tsx`
- `src/app/(public)/reset-password/actions.ts`

### 実装ポイント

#### 1. page 側

- `source=admin|clinic` を path segment で受ける
- 初期表示でセッション存在を確認する
- 有効ならフォーム表示
- 無効なら「リンク無効 / 期限切れ」状態UIを表示

#### 2. action 側

- `getServerClient()` で recovery セッション付きクライアントを取得する
- `passwordRecoverySchema` で検証する
- `supabase.auth.updateUser({ password })` を呼ぶ
- 成功後は `supabase.auth.signOut()` する
- `source` に応じて戻り先を分ける
  - `admin` -> `/admin/login?message=password_reset_completed`
  - `clinic` -> `/login?message=password_reset_completed`

#### 3. エラーメッセージ

- セッション不在
- 期限切れ
- パスワード不一致
- セキュリティポリシー不一致

を区別して表示する

#### 4. 監査ログ

- `password_reset_completed`
- `password_reset_invalid_link`

を記録する

### 完了条件

- recovery セッション時にパスワード更新できる
- 無効リンク時のUIが成立する
- 成功後はログイン画面に戻れる

### 検証

- integration:
  - 有効セッションでの更新成功
  - 無効セッション
  - confirm mismatch
- component:
  - invalid link state

## PR-05 `/login` `/admin/login` 導線追加 + rate limit 追加

### 目的

- 既存ログイン画面から forgot-password に遷移できるようにし、レート制限にも含める。

### 変更対象

- `src/app/(public)/login/page.tsx`
- `src/app/(public)/admin/login/page.tsx`
- `src/lib/rate-limiting/middleware.ts`

### 実装ポイント

#### 1. forgot password 導線追加

- スタッフログイン:
  - `/forgot-password?source=clinic`
- 管理者ログイン:
  - `/forgot-password?source=admin`

#### 2. レート制限対象追加

`isAuthEntryPoint()` に以下を追加する。

- `/forgot-password`
- `/reset-password`

`/reset-password/[source]` は prefix 判定へ変更した方が安全。

### 完了条件

- 両ログイン画面に forgot password 導線が出る
- forgot/reset が auth entry point として扱われる

### 検証

- 手動: 各ログイン画面から forgot-password 遷移
- unit: `isAuthEntryPoint()` の path 判定

## 6. 既知の注意点

| 項目 | 場所 | 内容 |
| --- | --- | --- |
| Auth メール前提 | Supabase Dashboard / Resend | SMTP 未設定だと recovery メールは送れない |
| `AuthResponse` 非適合 | `src/lib/schemas/auth.ts` | `confirmPassword` を扱えないため recovery 専用型が必要 |
| `/reset-password` 公開ルート | middleware | protected prefix には含まれていないため、画面自体は公開のままでよい |
| callback 依存 | `src/app/(public)/admin/callback/route.ts` | `next` 契約を壊さないことが重要 |
