# Tiramisu 売上文脈・care episode・療養費見込み分析 仕様書 v0.4

## 0. v0.4での更新判断

本仕様は v0.3 の「daily_report_items を日報明細SSOTとして拡張する」方針を維持する。

v0.4では、以下を追加・修正する。

1. Supabase Custom OAuth/OIDC Providers 対応をAuth/エンタープライズ導入章として追加
2. Supabase ISO/IEC 27001:2022 認証をセキュリティ・調達説明章に追加
3. `revenue_context_code` の既存データバックフィル順を修正
4. arrived予約トリガーで `revenue_context_code` / `amount_source` を明示更新する
5. 新規集計viewに `security_invoker = true` を必須化
6. 新規子テーブルにtenant整合性triggerを追加
7. Phase 1/2/3を別マイグレーションに分割
8. TypeScript型・hook・UIの更新対象を明示

## 1. 現行リポジトリ前提

現行リポジトリでは、売上・日報の実務中心は `daily_report_items` である。

```txt
reservations
  ↓ arrived trigger
daily_reports
  ↓
daily_report_items
  ↓
revenue aggregation
```

`daily_report_items.billing_type` は現状 `insurance | private` の2値であり、既存の `daily_reports.insurance_revenue` / `daily_reports.private_revenue` 集計と `/api/revenue` の互換性を持っている。

したがって、`billing_type` を多値化しない。

採用方針は以下。

| 項目                           | 役割                           |                |
| ---------------------------- | ---------------------------- | -------------- |
| `billing_type`               | 既存互換用。`insurance             | private` のまま維持 |
| `revenue_context_code`       | 新しい分析主分類。保険/自費/事故/労災/物販/回数券等 |                |
| `fee`                        | 現在の認識売上・日報集計金額               |                |
| `revenue_estimates`          | 療養費/売上見込みの計算結果スナップショット       |                |
| `revenue_estimate_lines`     | 計算内訳                         |                |
| `revenue_estimate_warnings`  | 要確認・除外・制度上の注意                |                |
| `revenue_estimate_overrides` | 手動補正履歴                       |                |

## 2. Supabase 2026アップデート反映

### 2.1 Custom OAuth/OIDC Providers の位置づけ

Supabase Auth は Custom OAuth/OIDC Providers を追加した。

Tiramisuでは、初期MVPでは必須実装にしない。ただし、将来的なグループ院・法人導入・医療/準医療系ネットワーク導入に備えて、Auth設計に拡張余地を残す。

対象ユースケース。

| ユースケース                                  | 価値            | 初期実装判断            |
| --------------------------------------- | ------------- | ----------------- |
| Google Workspaceログイン                    | 小規模院・個人院向け    | 既存/通常OAuthで十分     |
| LINEログイン                                | 患者向けLIFF/予約導線 | 別設計で維持            |
| GitHub Enterprise / GitLab self-managed | 開発者・社内管理者向け   | 今は不要              |
| 地域/業界固有IdP                              | 公的/業界ネットワーク対応 | 将来オプション           |
| 企業SSO / SAML-to-OIDC bridge             | 大規模法人・本部導入    | Enterprise要件として保持 |
| 既存会員基盤との連携                              | 美容/治療院グループ横展開 | 将来オプション           |

### 2.2 Auth設計方針

Auth本体は以下のまま維持する。

```txt
Supabase Auth user
  ↓ JWT app_metadata
app_private.get_current_role()
app_private.get_current_clinic_id()
app_private.can_access_clinic()
  ↓
RLS tenant boundary
```

Custom OAuth/OIDC Providersを使う場合でも、RLSの正は外部IdPのclaimではなく、Tiramisu側の `profiles` / `user_permissions` / `clinic_scope_ids` に正規化したclaimとする。

つまり、外部IdPは「本人確認・ログイン入口」であり、院権限・本部権限・tenant境界の最終判断はTiramisu側で行う。

### 2.3 Custom Provider利用時の仕様

Custom Providerを利用する場合は以下を必須とする。

| 項目                    | 方針                                          |
| --------------------- | ------------------------------------------- |
| provider identifier   | `custom:<idp-name>` 形式                      |
| OIDC provider         | issuer URLを登録し、discovery/JWKSはSupabase側自動解決 |
| OAuth2 provider       | authorization/token/userinfo endpointを明示    |
| PKCE                  | 原則有効。無効化禁止に近い扱い                             |
| email optional        | 原則false。emailが無いIdPは別途ID linking設計が必要       |
| acceptable_client_ids | web/iOS/Android等でclient idが分かれる場合のみ使用       |
| authorization params  | consent/offline_access/login_hint等の明示用途だけ許可 |
| provider quota        | Free/Pro/Enterprise差異に注意。商用導入ではPro以上前提      |

### 2.4 Custom Provider導入時のDB拡張候補

初期MVPでは追加不要。

Enterprise対応時に以下を追加する。

```sql
create table if not exists public.auth_identity_provider_links (
  id uuid default extensions.uuid_generate_v4() not null,
  clinic_id uuid not null,
  user_id uuid not null,
  provider_identifier text not null,
  external_subject text not null,
  external_email text,
  external_claims jsonb not null default '{}'::jsonb,
  linked_at timestamptz not null default now(),
  last_seen_at timestamptz,
  created_at timestamptz not null default now(),

  constraint auth_identity_provider_links_pkey primary key (id),
  constraint auth_identity_provider_links_clinic_id_fkey
    foreign key (clinic_id) references public.clinics(id) on delete cascade,
  constraint auth_identity_provider_links_user_id_fkey
    foreign key (user_id) references auth.users(id) on delete cascade,
  constraint auth_identity_provider_links_provider_subject_unique
    unique (provider_identifier, external_subject)
);
```

このテーブルは、外部IdPのsubjectとTiramisu userを監査可能に紐づける用途。RLS権限判定の正にはしない。

### 2.5 Auth DoD

Custom OAuth/OIDC Providersを導入する場合のDoD。

* `app_private` helperがJWT `app_metadata` を正しく読む
* `role: authenticated` をアプリケーションロールとして扱わない
* 外部IdP claimをそのまま院権限に使わない
* `profiles.role` / `user_permissions.role` / `clinic_scope_ids` に正規化する
* provider secret rotation手順をdocsに残す
* provider削除時の既存ユーザー影響をdocsに残す
* E2Eで外部IdPログイン後、RLS境界が破れないことを検証する

## 3. Supabase ISO/IEC 27001:2022 反映

### 3.1 位置づけ

SupabaseがISO/IEC 27001:2022認証を取得したことにより、Tiramisuの営業・調達説明では以下を言える。

```txt
Tiramisuは、ISO/IEC 27001:2022認証済みのSupabase基盤上で、
PostgreSQL/RLS/Auth/Storage/Edge Functions等を利用して構築する。
```

ただし、これはTiramisu自体がISO 27001認証済みであることを意味しない。

### 3.2 営業・調達向け表現

使ってよい表現。

```txt
基盤として利用しているSupabaseは ISO/IEC 27001:2022 認証を取得しています。
Tiramisu側では、RLSによるtenant境界、監査ログ、権限管理、バックアップ/復旧手順を個別に整備します。
```

禁止表現。

```txt
TiramisuはISO 27001認証済みです。
医療情報管理として完全準拠しています。
請求・療養費・個人情報管理はすべて保証済みです。
```

### 3.3 本仕様での追加要件

本仕様のDB/API追加に対し、以下を必須にする。

* 新規テーブルはRLS有効化
* clinic scoped tableは `clinic_id` を必須保持
* viewは `security_invoker = true`
* 手動補正は上書きではなく履歴保存
* revenue estimateにはdisclaimerを持たせる
* auditに必要な `created_by` / `updated_by` / `created_at` / `updated_at` を可能な限り保持
* 請求確定額ではなく経営分析用概算として表示

```sql
returns trigger
language plpgsql
set search_path = public, auth, extensions
as $$
declare
  v_item_clinic_id uuid;
begin
  select clinic_id
  into v_item_clinic_id
  from public.daily_report_items
  where id = new.daily_report_item_id;

  return new;
end;
```

## 8. 集計view

新規viewは必ず `security_invoker = true`。

### 8.1 daily_report_revenue_context_summary

```sql
create or replace view public.daily_report_revenue_context_summary
with (security_invoker = true)
as
select
  dri.clinic_id,
  dri.report_date,
  dri.revenue_context_code,
  rc.name as revenue_context_name,
  rc.rollup_category,
  count(*)::integer as item_count,
  coalesce(sum(dri.fee), 0)::numeric(10,2) as total_revenue,
  count(*) filter (where dri.estimate_status = 'needs_review')::integer as needs_review_count,
  count(*) filter (where dri.estimate_status = 'blocked')::integer as blocked_count
from public.daily_report_items dri
join public.revenue_contexts rc
  on rc.code = dri.revenue_context_code
group by
  dri.clinic_id,
  dri.report_date,
  dri.revenue_context_code,
  rc.name,
  rc.rollup_category;
```

### 9.2 互換ルール

```ts
function deriveLegacyBillingType(
  revenueContextCode: RevenueContextCode
): BillingType {
  return revenueContextCode === 'insurance' ? 'insurance' : 'private';
}

function deriveRevenueContextCodeFromBillingType(
  billingType: BillingType

  return billingType === 'insurance' ? 'insurance' : 'private';
```

`billingType` と `revenueContextCode` が同時に来て矛盾する場合は400。

```txt
billingType=private + revenueContextCode=insurance -> reject
```

5. 明細を `traffic_accident` に変更
6. `TRAFFIC_ACCIDENT_REVIEW` tagを付ける
7. 本部 revenue context summary に交通事故売上が出る
8. 他clinicユーザーから見えない

## 12. Phase 2: care episode / visit stage

v0.3方針を維持する。

ただし、Phase 1とは別マイグレーションにする。

```sql
create table if not exists public.care_episodes (...);
create table if not exists public.visit_stage_definitions (...);

alter table public.daily_report_items
  add column if not exists care_episode_id uuid,
  add column if not exists visit_ordinal_in_episode integer,
  add column if not exists visit_stage_code text;
```

`validate_daily_report_items_analysis_refs()` はPhase 2で導入する。

## 13. Phase 3: revenue estimates

v0.3方針を維持する。

ただし、子テーブルtenant整合性triggerを必須にする。

対象。

```txt
revenue_estimates
revenue_estimate_lines
revenue_estimate_warnings
revenue_estimate_overrides
```

共通方針。

* `clinic_id` 必須
* 親レコードの `clinic_id` と一致必須
* RLS必須
* 手動補正は履歴保存
* 請求確定に見せない

UI文言。

```txt
経営分析用の概算です。請求確定額ではありません。
```

## 14. 絶対に避けること

1. `patients` を新規主役にしない
2. `billing_type` を多値化しない
3. `public.can_access_clinic()` で新規RLSを書かない
4. `daily_reports` に大量集計カラムを先に足さない
5. `revenue_context_code text not null default 'private'` を既存データに直接入れない
6. `security_invoker = true` なしでviewを作らない
7. 外部IdP claimをそのまま院権限に使わない
8. SupabaseのISO 27001をTiramisu自体の認証のように表現しない
9. 療養費見込みを請求確定額に見せない

## 15. 最終判断

Phase 1は即実装してよい。

ただし、v0.3のままでは `revenue_context_code` のバックフィル順が危険で、保険行をprivateに誤分類する可能性がある。v0.4ではこの順序を修正した。

Supabase Custom OAuth/OIDC Providers は、今すぐMVPに入れる機能ではない。しかし、将来の大規模院・法人・業界ネットワーク・SSO導入のためにAuth設計上の拡張余地として仕様に含める。

Supabase ISO/IEC 27001:2022 は営業・調達上の信用補強として有効。ただし、Tiramisu自体の認証ではないため、表現は厳密に分ける。

結論。

```txt
daily_report_items を日報明細SSOTとして拡張
↓
revenue_context_code で売上文脈を分類
↓
view/API/UIで本部分析価値を先に出す
↓
care episode / visit stage はPhase 2
↓
療養費見込み計算はPhase 3
↓
Enterprise Auth/ISO説明は商談・調達対応の補助線として保持
```
