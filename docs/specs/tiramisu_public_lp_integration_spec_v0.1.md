# Tiramisu 公開LP統合仕様書 v0.1

## 目的

`TiramisuLP` にある既存LPデザインを、公開トップ `/` として `src/app/(public)/page.tsx` に統合する。

今回の目的は LP を新規デザインすることではなく、完成済みのLP案を現行 Next.js アプリの公開ルート、認証導線、料金設計、検証フローに合わせて安全に載せることである。

料金と訴求は、マーケットチーム案に合わせて「5店舗以上の整骨院グループ向け本部管理OS」を主軸にする。単店舗向けの見せ方は残してもよいが、主役にはしない。

## 判断

`TiramisuLP` のデザインは採用ベースで進める。

理由:

- Hero、課題提起、機能訴求、AIデモ、料金、導入導線、登録フォーム、FAQ、フッターまでLPとして一通り揃っている
- デスクトップとスマホの表示で大きなレイアウト崩れは見えない
- 色味、余白、カード設計、タイポグラフィは外部共有用LPとして成立している
- 現在の `/` は簡易ログイン入口であり、サービス説明・需要検証・営業共有の受け皿として弱い

ただし、`TiramisuLP` をそのまま丸ごと移植しない。現行プロダクトと公開ルート仕様に合わせて、情報・導線・実装方式を調整する。

## 前提

### 現行アプリ側

- `origin/main` には公開面と認証後面の route group 分離が入っている
- 公開トップは `src/app/(public)/page.tsx`
- 公開ログイン導線は以下を維持する
  - `/login`
  - `/admin/login`
  - `/terms`
  - `/privacy`
- `middleware.ts` では `/login` と `/admin/login` が公開ルートとして扱われ、業務画面は保護されている
- パッケージマネージャーは npm
- 新規依存は追加しない

### 既存LP側

- `TiramisuLP` は Vite + React の独立プレビュー
- メイン実装は `TiramisuLP/tiramisu-lp-v3.jsx`
- `index.html` は Tailwind CDN を読み込む
- `GLOBAL_STYLES` 内で Google Fonts を import している
- `lucide-react` を利用している
- 先行登録フォームは `VITE_SHEETS_ENDPOINT` 前提
- AIデモは `/api/gemini` への fetch 前提
- 法務ページは hash state によるLP内ページとして実装されている

## スコープ

### 対象

- `src/app/(public)/page.tsx` のLP化
- 必要に応じた公開LP用コンポーネント分割
- LP内ヘッダーまたはフッターへの `/login` と `/admin/login` 導線追加

## 過不足の洗い出し

### 0. 価格・プランの取捨選択

#### 採用する基本方針

公開LPでは、Tiramisuを単店舗向けの安価な予約・日報ツールとして見せない。

採用する訴求:

- 5店舗以上の整骨院グループ向け本部管理OS
- 日報、売上、予約、シフト、店舗比較を一元管理する管理パッケージ
- 本部業務の集計・確認・分析を減らすクラウド
- 多店舗経営の数字を毎日見える化するサービス

避ける訴求:

- 本部機能: 月額 50,000円
- 店舗追加: 月額 8,000円
- 予約管理システム
- 日報ツール
- レセコンの代替

#### 公開LPに前面掲載するプラン

料金セクションでは、以下の3プランを主表示にする。

| プラン | 月額 | 含まれる店舗数 | 初期費用 | LPでの位置づけ |
|---|---:|---:|---:|---|
| Group Starter | 78,000円 | 5店舗まで | 300,000円〜 | 導入しやすい入口 |
| Group Standard | 128,000円 | 10店舗まで | 300,000円〜500,000円 | 主力プラン |
| Enterprise | 198,000円〜 | 20店舗以上 | 500,000円〜 | 大型法人向け |

追加店舗の表記:

- Group Starter: 6店舗目以降 8,000円/月
- Group Standard: 11店舗目以降 8,000円/月
- Enterprise: 個別見積

LP上の強調:

- Group Standard を「おすすめ」または「主力」として最も目立たせる
- Group Starter は「まず本部集計を整えたい法人向け」として入口に置く
- Enterprise は「外部連携、独自帳票、運用設計が必要な法人向け」として問い合わせ導線にする

#### 補助的に掲載する情報

Single Plan は存在を消さず、ただし前面には出さない。

掲載候補:

- 料金セクション下の小さな補足
- FAQ
- 問い合わせ前提の注記

表記例:

- 単店舗向けの簡易プランは月額 12,000円〜。詳細はお問い合わせください。

年払いと補助金・法人稟議向けパッケージは、初回LPでは詳細テーブルを出しすぎない。

掲載候補:

- FAQで「年払い、補助金活用、法人稟議向けの見積に対応可能」と書く
- CTA付近で「初年度導入パッケージは個別にご相談ください」と補足する

#### 公開LPに載せない情報

以下は営業内部向けの判断材料であり、初回公開LPには載せない。

- 「単院はLTVが低い」「積極的には売らない」などの内部判断
- 店舗数別の詳細な価格シミュレーション表
- 100店舗など大型すぎる価格目安
- 年払いの詳細比較表
- 補助金パッケージの詳細内訳
- 初期費用の全オプション価格表
- 先行割引、登録数、ローンチ月など未確定の訴求

ただし、営業資料や個別提案書では使える情報なので、LPから削るだけで企画上は保持する。

### 2. 不足しているもの

#### 既存利用者向けログイン導線

`TiramisuLP` には現行アプリのログイン導線が不足している。

必須対応:

- ヘッダーに `スタッフログイン` → `/login`

- フッターに `/terms` と `/privacy`

#### マーケットチーム料金案との整合

`TiramisuLP` の料金は以下のような表記になっている。

- 単院プラン: 月額 12,000円
- 多店舗プラン: 月額 100,000円
- 先行登録 30% off など

以前の統合案には以下の古い料金前提が残っている。

- 本部機能: 月額 50,000円
- 1店舗追加: 月額 8,000円

今回のマーケットチーム案では、以下を公開LPの料金前提にする。

- Group Starter: 月額 78,000円、5店舗まで込み、6店舗目以降 8,000円/月、初期設定費 300,000円〜
- Group Standard: 月額 128,000円、10店舗まで込み、11店舗目以降 8,000円/月、初期設定費 300,000円〜500,000円
- Enterprise: 月額 198,000円〜、20店舗以上、個別見積、初期設定費 500,000円〜
- Single Plan: 月額 12,000円〜、単店舗向け簡易プランとして補助的に扱う

必須対応:

- 初回統合ではマーケットチーム案の料金に合わせる
- 「本部機能 50,000円」という旧表現は使わない
- Group Standard を主力プランとして目立たせる
- Single Plan は料金カードの主役にしない
- 先行割引、登録数、ローンチ月などは確定していない場合は出さない
- 税抜/税込が未確定なら「税別」などを安易に明記しない

#### CTA の行き先

`TiramisuLP` は先行登録フォームと `mailto:founder@tiramisu.clinic` を持つ。初回LPでは、本体側に問い合わせAPIを作らず、Google Form への外部リンクを正式な受付導線にする。

必須対応:

- 初期版では CTA を「デモ相談」「資料請求」「問い合わせ」に整理する
- CTA の主導線は Google Form に遷移させる
- LP内に本番送信フォームは実装しない
- Google Form の URL は確定後に設定する
- URL 未確定の間は、実装上は定数に仮値を置かず、CTA を `#contact` などの説明セクションに向ける
- `mailto:` は Google Form が使えない場合の補助導線に留める
- Google Form 側で収集する項目、個人情報の扱い、返信体制を運用側で確定する

#### Next.js 統合方式

`TiramisuLP` は Vite と Tailwind CDN 前提であり、現行 Next.js へ直接貼るには不足がある。

必須対応:

- Tailwind CDN を使わない
- `GLOBAL_STYLES` の Google Fonts import をそのまま持ち込まない
- `import.meta.env` を使わない
- `.jsx` ではなく TypeScript/TSX として型安全に移す
- `any`、`as any`、`@ts-ignore` を使わない

#### 公開ルートの法務ページ整合

`TiramisuLP` は `#privacy-policy`、`#terms`、`#commerce-law` をLP内ページとして持っている。

必須対応:

- 既存の `/terms` と `/privacy` を正とする
- LP内 hash 法務ページは初回統合では持ち込まない
- 特商法ページが必要なら別途 `/commerce` などの仕様を切る

#### SEO / metadata

現行の簡易トップにはLP用の metadata が不足している。

必要対応:

- `title`
- `description`
- OGP は別タスクでもよいが、最低限の文言方針は決める

### 3. 過剰なもの

#### 本番AIデモ

`TiramisuLP` には `/api/gemini` に問い合わせるAIデモがある。

初回統合では本番AIとして動作させない。代わりに、選択肢クリック式のモックAIデモとして実装する。

理由:

- APIキー、レート制限、ログ、個人情報入力、エラー表示の設計が別途必要
- LP公開時に未保護APIを増やすリスクがある
- 自由入力にすると、実AIとしての期待値や個人情報入力リスクが上がる
- デモとして見せるだけなら、質問選択式と回答プールで十分

対応:

- AIデモは選択肢クリック式のモック表示として採用する
- 選択肢は導入候補者が見たい観点に絞る
  - 昨日の売上が落ちた理由
  - キャンセル率が高い店舗
  - スタッフ別の改善ポイント
  - 予約枠の空き状況
  - 今月の本部レポート要約
- クリックされた選択肢に応じて、事前定義した複数の回答候補からランダムに1つ返す
- 回答は「実データ分析結果」ではなく「分析イメージ」として表現する
- 回答内で数値を出す場合は、架空データであることが分かる文脈にする
- 入力欄から実API送信するUIは初回では置かない
- 入力欄から実API送信する機能は別PRに分離する

#### Google Form 導線

`VITE_SHEETS_ENDPOINT` 前提のフォーム送信は初回統合では無効化する。問い合わせ、デモ相談、資料請求は Google Form への外部リンクで受ける。

理由:

- Next.js 側の環境変数名、送信先、CORS、スパム対策、個人情報保護の確認が必要
- 公開フォームを独自実装すると運用責任が発生する
- Google Form であれば初期運用、通知、回答管理を早く始められる
- LP初回PRではフォームAPI、DB保存、スパム対策を持ち込まなくて済む

対応:

- 初期版は Google Form への外部リンクを正式導線にする
- CTA は同一の Google Form に向けてもよい
  - デモ相談
  - 資料請求
  - 問い合わせ
- Google Form 側で問い合わせ種別を選べるようにする
- LP内に送信完了状態や擬似送信UIは作らない
- 将来的に自社フォームへ移行する場合は別PRに分離する

#### LP内の複数ページ管理

`currentPage` と hash route で `home`、`aiChat`、`preRegister`、`developerContact`、`privacy`、`terms`、`commerce` を切り替えている。

初回統合では過剰。

対応:

- `/` は単一LPとして実装する
- 詳細ページ化が必要なものは Next.js route として別途切る

#### 不確定な社会的証明

`registeredCount`、`monthlyIncrease`、`prefectureCount`、`launchMonth` などは、事実確認なしに本番LPへ出さない。

対応:

- 確定値がないものは削除または抽象表現にする
- 「先行登録 83院」などは証跡がある場合のみ表示する

## 実装方針

### ファイル方針

第一候補:

- `src/app/(public)/page.tsx`

ページが大きくなりすぎる場合:

- `src/app/(public)/page.tsx`
- `src/components/public/tiramisu-landing-page.tsx`

LP用の定数を分離する場合:

- `src/components/public/tiramisu-landing-content.ts`

### 技術方針

- 既存の Next.js App Router に合わせる
- `Link` は `next/link` を使う
- 既存依存の `lucide-react` は利用可
- 新規依存は追加しない
- Tailwind CDN は使わない
- CSS は既存 Tailwind class と必要最小限の local class に寄せる
- 外部画像を使う場合は `next/image` の remote 設定が必要になるため、初回は使わないかCSS/既存要素で代替する
- `import.meta.env` は使わない
- API送信やフォーム送信は初回では実装しない
- Google Form URL はLP用定数として管理する
- Google Form URL 未確定時に、空文字や仮URLで公開しない
- AIチャットデモはローカル状態だけで完結させる
- AIチャットデモのランダム応答は `Math.random()` で十分だが、回答候補は選択肢ごとに型付き配列で管理する

### TypeScript 方針

- `any` は使わない
- `as any` は使わない
- `Record<string, any>` は使わない
- CTA、FAQ、機能カード、料金カードは明示的な型を定義する
- AIチャットデモの選択肢と回答候補も明示的な型を定義する
- クライアント状態が不要なセクションは Server Component として実装する
- AIチャットデモ、FAQアコーディオン、sticky CTA が必要な場合だけ Client Component を分離する

## セキュリティ / 認可

- LPは公開ページであり、認可の代替にしない
- 業務データ、患者データ、予約データへアクセスしない
- Supabase client をLPから呼ばない
- Gemini、Google Sheets、問い合わせ送信APIは初回統合の対象外
- Google Form への遷移は外部リンクとして扱い、LP側では問い合わせ内容を保存しない
- AIチャットデモは業務データ、患者データ、予約データを扱わない
- `/login` と `/admin/login` の公開性は現行 middleware に委ねる
- RLS、tenant isolation、clinic scope は変更しない

## 結論

`TiramisuLP` はLPデザインとして採用できる。ただし現行本体へ統合するには、料金、ログイン導線、法務導線、AIデモ、フォーム導線、外部依存を調整する必要がある。

料金は旧案の「本部機能 50,000円」ではなく、マーケットチーム案の `Group Starter`、`Group Standard`、`Enterprise` を主軸にする。公開LPでは `Group Standard` を本命として見せ、`Single Plan` は補助的に扱う。

初回は「完成済みLPデザインを公開トップに据える」ことに集中する。AIチャットデモは選択肢クリック式のモックにし、問い合わせは Google Form への外部リンクで受ける。これにより、外部共有可能なトップページを早く出しつつ、認証・RLS・業務画面への影響を最小化できる。
