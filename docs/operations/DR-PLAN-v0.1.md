# 災害復旧計画（DR）v0.1

## 目的と保証範囲

限定ベータの内部運用目標は次のとおりです。

| 指標 | 内部目標   | 定義                                                           |
| ---- | ---------- | -------------------------------------------------------------- |
| RTO  | 8時間以内  | 業務中断を確認してから、主要な認証・参照・予約業務を再開できるまで |
| RPO  | 24時間以内 | 復旧時点から遡って失われる可能性があるデータの最大時間         |

これらは設計・訓練のための非契約的な内部目標であり、SLA、返金条件、可用性保証ではありません。Supabase の契約プラン、バックアップ設定、障害範囲、復元テストの結果によって達成できない場合があります。

## 商用開始前のブロッカー

次の証跡がそろうまで、RTO/RPO 達成やバックアップ完備を外部へ表明しません。

- [ ] 本番 Supabase プランで利用できるバックアップ方式を確認した
- [ ] 保持期間、復元単位、復元先、リージョン制約を記録した
- [ ] Storage を含む対象データと対象外データを棚卸しした
- [ ] Vercel と Supabase の設定・環境変数の復旧手順を確認した
- [ ] 隔離環境への復元訓練を実施し、RTO/RPO の実測値を記録した
- [ ] 復元後のテナント分離・認可テストを完了した

## 対象

- アプリケーション: Vercel のデプロイと設定
- データベース・認証・Storage: Supabase プロジェクト
- 構成情報: 環境変数名、ドメイン、外部サービス設定、運用連絡網
- 証跡: 監査ログ、インシデント記録、復元訓練記録

シークレット値は Runbook、Issue、チャット、スクリーンショットへ保存しません。

## 復旧判断

復旧開始前に以下を記録します。

1. 障害の開始時刻と影響範囲
2. 最新の安全な復元ポイント
3. 復元によって失われる可能性がある更新の期間
4. 復元先と切替方式
5. 実行者、承認者、利用者連絡担当
6. 復旧を中止する条件

## 復旧手順

1. **宣言**: インシデント指揮者を決め、記録テンプレートを開始する
2. **封じ込め**: 必要な場合だけ書き込み停止や公開停止を行い、実施内容を記録する
3. **復元ポイント選定**: プロバイダー画面で利用可能な時点を確認し、想定データ損失を承認者へ提示する
4. **隔離復元**: 可能な限り新しい隔離プロジェクトまたは検証環境へ、プロバイダーが提供する手順で復元する
5. **検証**: スキーマ整合、重要テーブル件数、最新更新時刻、認証、院スコープ、RLS、主要APIを確認する
6. **切替承認**: 検証結果と残存リスクを記録し、本番切替の承認を得る
7. **切替と監視**: アプリ設定を切り替え、ヘルスチェックとロール別スモークテストを実施する
8. **完了判定**: 利用者影響、失われた可能性のあるデータ、再入力が必要な範囲を確定する
9. **事後対応**: インシデント記録、顧客連絡、再発防止策を更新する

復元、プロジェクト切替、シークレットローテーションは状態変更を伴います。実環境で行う前に明示的な承認を取得してください。

## 復元後の最低検証

- [ ] 管理者、院管理者、スタッフの認証が期待どおり動作する
- [ ] 別院の患者・予約・日報を参照または更新できない
- [ ] 患者、予約、日報の件数と最新更新時刻が選定した復元ポイントと整合する
- [ ] 予約の参照・作成・変更で重複や欠落がない
- [ ] 監査ログとエラーログが記録される
- [ ] 通知・決済など外部副作用の二重実行がない

## 訓練

初回商用利用前と、その後少なくとも四半期ごとを内部訓練目標とします。契約プランや構成変更、重大障害後にも再実施します。実施できなかった場合は理由、リスク、次回予定を記録し、実績があるように表現しません。

訓練記録には [復元訓練記録テンプレート](./RESTORE_DRILL_TEMPLATE.md) を使用します。
