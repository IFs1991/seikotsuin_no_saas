# 本番・限定ベータ運用 Runbook v0.1

## 文書の位置づけ

この文書は、限定ベータで障害を検知し、影響を抑え、安全に復旧判断を行うための内部手順です。
記載する時間は契約上の SLA、可用性保証、24時間365日の有人対応を約束するものではありません。
担当者、連絡先、対応時間は運用開始前に安全な連絡網へ登録し、このリポジトリには個人情報を記載しません。

## 運用開始前の必須確認

- [ ] インシデント指揮者、技術担当、顧客連絡担当と代替担当を連絡網へ登録した
- [ ] 対応可能時間と連絡手段を顧客向け条件と矛盾しない形で確定した
- [ ] Vercel と Supabase の本番プロジェクト、契約プラン、リージョン、バックアップ設定を記録した
- [ ] バックアップ保持期間と復元可能な方式をプロバイダー画面で確認した
- [ ] 本番 URL と監視通知先を確定した
- [ ] `docs/operations/RESTORE_DRILL_TEMPLATE.md` を使った復元訓練を完了した
- [ ] 利用規約・プライバシーポリシーを事業者情報入りで専門家レビューした

未確認項目がある場合、バックアップ、復旧時間、法令適合、認証取得を顧客へ保証してはいけません。

## 日次確認（PowerShell）

シークレット値は画面やログへ出力しません。

```powershell
$AppUrl = $env:APP_URL
if ([string]::IsNullOrWhiteSpace($AppUrl)) {
  throw 'APP_URL is required'
}

$health = Invoke-RestMethod -Method Get -Uri "$AppUrl/api/health"
$health | ConvertTo-Json -Depth 6

vercel logs --since 24h --level error --json
```

確認結果は日時、実行者、環境、結果、関連チケットだけを運用記録へ残します。レスポンスやログに患者情報、認証トークン、シークレットが含まれる場合は転記しません。

## デプロイ前後

### デプロイ前

リポジトリで定義済みの npm スクリプトを使用します。

```powershell
npm run lint
npm run type-check
npm run test -- --ci --testPathIgnorePatterns=e2e
npm run build
```

- CI の必須チェックが成功していることを確認する
- 変更対象、ロールバック候補、データ変更の有無を記録する
- マイグレーションを伴う場合は、別途承認済みの仕様書とロールバック計画が必要

### デプロイ後

- `/api/health` と主要画面の読み取りを確認する
- 対象ロールで認証、院スコープ、書き込み可否を確認する
- 別院データを参照できないことを確認する
- エラーログ、通知処理、決済を変更した場合は該当フローを確認する

## インシデント初動

1. インシデント記録を作成し、指揮者を1名決める
2. 発生時刻、検知方法、影響する院・機能・データを記録する
3. 証拠を保全する。ログや監査記録を削除・改変しない
4. 影響拡大を防ぐ最小の可逆対応を選ぶ
5. データ変更、認証情報ローテーション、復元、書き込み停止は承認者を記録してから行う
6. 顧客連絡は確認済みの事実、現在の影響、次回更新時刻だけを伝える

セキュリティ侵害が疑われる場合は、シークレットを表示せず、各プロバイダーの管理画面で無効化・再発行します。対象、実行者、時刻をインシデント記録へ残します。

## アプリケーションの安全なロールバック

推奨順序:

1. Vercel Dashboard で直前の正常なデプロイを特定する
2. 差分とデータ互換性を確認する
3. 承認後、直前の正常デプロイを Promote する
4. ヘルスチェックと主要ロールのスモークテストを実施する

コードで取り消す場合は、新しい revert commit を作成して通常のレビューと CI を通します。

```powershell
git revert <commit-hash>
git push origin <branch-name>
```

`git reset --hard`、force push、履歴の書き換えを通常の復旧手順に使用しません。

## データ復元

データ復元は通常運用ではなく状態変更を伴う緊急作業です。[DR 計画](./DR-PLAN-v0.1.md) に従い、復元ポイント、失われる可能性がある更新、承認者を確定してから実施します。

- 本番へ直接、未検証の SQL や復元操作を実行しない
- 可能な限り隔離した復元先で検証する
- テナント境界、認証、件数、最新更新時刻、重要業務フローを確認する
- 復元結果を「成功」と記録する前に、利用者側の確認を含む完了条件を満たす

## DoD との対応

この Runbook は `docs/stabilization/DoD-v0.1.md` の DOD-01、DOD-06、DOD-08、DOD-10、DOD-11 を本番運用へつなぐ補助資料です。Runbook の存在だけでは各 DoD や商用準備の達成証跡にはなりません。実行日時と結果を別途記録してください。
