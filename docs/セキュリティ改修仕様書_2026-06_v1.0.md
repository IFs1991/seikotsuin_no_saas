## 0. 概要

本仕様書は、セキュリティ監査で検出された脆弱性・ハードニング項目を改修するための実装仕様を定義する。

監査の総評として、**コア設計（RLSによるテナント分離、JWTクレーム駆動の認可、シークレット管理、Docker/CI）はエンタープライズ水準で堅牢**であり、インジェクション系（SQL/PostgREST/XSS/SSRF/コマンド注入/オープンリダイレクト）に Critical/High は存在しない。

一方で「**立派な防御機構が実装されているのに、実際には作動していない／迂回できる**」というパターンの重大な穴が複数存在する。本改修はこれらを塞ぐことを目的とする。

### SEC-02 🔴 CSPの本番強制化と nonce 配線

#### 問題
1. **dev CSPが本番で強制されうる**: `middleware.ts:148` のデフォルトが `'report-only'`。`csp-config.ts:408-413` の `report-only` 分岐は**強制ヘッダ** `Content-Security-Policy` に `getDevelopmentCSP()` を設定する。これは `script-src 'unsafe-inline' 'unsafe-eval'`（`csp-config.ts:37-43`）を含むため、XSSが発生した瞬間にCSPが無力になる。厳格ポリシーは `Content-Security-Policy-Report-Only`（ブラウザは強制しない）でしか送られない。
2. **nonce未消費**: `middleware.ts:138-139` で per-request nonce を生成し `x-nonce` を**レスポンス**ヘッダに設定するが、どのlayout/componentも読まない。Next.js が nonce を適用するには**リクエスト**ヘッダ経由で伝播させる必要があり、現状の配線では厳格な nonce ベースCSPが機能しない。

#### 該当箇所
- `middleware.ts:138-162`
- `src/lib/security/csp-config.ts:34-69`（dev CSP）, `:401-428`（rollout）, `:391-396`（fallbackで `'unsafe-inline'` に劣化）
- `.env.local.example:24`, `.env.test.example:62`（`report-only`）, `.env.production.example:21`（`full-enforce` ※本番例は正しい）

#### 攻撃シナリオ
本番運用者が `CSP_ROLLOUT_PHASE=full-enforce` を明示設定しない限り、強制CSPはdev用（`unsafe-inline`/`unsafe-eval`許可）となる。反射型/格納型XSSのペイロードがそのまま実行される。

#### 改修内容

**(1) ミドルウェアのデフォルトを本番では `full-enforce` に**
```ts
// middleware.ts
const defaultPhase =
  process.env.NODE_ENV === 'production' ? 'full-enforce' : 'report-only';
const phaseEnv =
  (process.env.CSP_ROLLOUT_PHASE as
    | 'report-only'
    | 'partial-enforce'
    | 'full-enforce'
    | undefined) ?? defaultPhase;
```

**(2) nonce をリクエストヘッダへ伝播し、root layout で読む（Next.js App Router 公式パターン）**
```ts
// middleware.ts — createNextResponse 生成時に request ヘッダへ nonce を載せる
const requestHeaders = new Headers(request.headers);
requestHeaders.set('x-nonce', nonce);

const response = NextResponse.next({ request: { headers: requestHeaders } });
// 既存のレスポンス側ヘッダ設定（セキュリティヘッダ等）は維持
```
```tsx
// src/app/layout.tsx（root layout）
import { headers } from 'next/headers';

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const nonce = (await headers()).get('x-nonce') ?? undefined;
  // next/script を使う場合は nonce を渡す。Next.js は CSP ヘッダ中の nonce を
  // 自前のブートストラップ/ハイドレーションscriptに自動適用する。
  return (
    <html lang="ja">
      <body>{children}</body>
    </html>
  );
}
```

**(3) 厳格CSPに `'strict-dynamic'` を付与**
`getMedicalGradeCSP()` / `getProductionCSP(nonce)` の `script-src` を `'self' 'nonce-<nonce>' 'strict-dynamic'` とし、nonce付きローダーから読み込まれる後続scriptを許可する。これによりスクリプトホストの個別許可リストが不要になり、Nextのhydrationが壊れない。

**(4) fallbackの劣化を防止**
`csp-config.ts:391-396` `getFallbackStyleHashes()` が `'unsafe-inline'` を返している。`style-src` も nonce ベース（または最小化したハッシュ）に統一し、**エラー時に `'unsafe-inline'` へ劣化しない**ようにする。静的style-hashが陳腐化している場合はビルド時生成へ寄せる。

**(5) report-only分岐の是正**
本番が `full-enforce` をデフォルトにするため、`report-only` は開発/検証専用となる。`report-only` 分岐で**強制ヘッダにdev CSPを設定する挙動**は開発限定の許容とし、本番経路から到達しないことをテストで保証する。

#### 受け入れ条件
- [ ] 本番ビルド（`NODE_ENV=production`、`CSP_ROLLOUT_PHASE` 未設定）で `Content-Security-Policy` ヘッダが厳格ポリシー（`unsafe-inline`/`unsafe-eval` を**含まない**）になる。
- [ ] 当該ヘッダに `'nonce-...'` と `'strict-dynamic'` が含まれる。
- [ ] アプリが正常にハイドレーションし、コンソールにCSP違反が出ない（主要画面: login / dashboard / admin / reservations）。
- [ ] SEC-01 完了済み（nonce XSS advisory修正済みのNext版）。

#### テスト
- `src/__tests__/security/` に CSP ヘッダ検証テストを追加：
  - `getGradualRolloutCSP('full-enforce', nonce).csp` に `unsafe-inline`/`unsafe-eval` が**含まれない**こと。
  - nonce が毎回ユニークであること（`generateNonce()` 連続呼び出しで不一致）。
  - fallback style が `unsafe-inline` を返さないこと。
- E2E（Playwright）で主要画面のCSP違反ゼロを確認。

### SEC-03 🟠 レート制限のIPスプーフィング対策と fail-closed 化

#### 問題
1. **IPスプーフィング**: `src/lib/rate-limiting/middleware.ts:278-298` の `getClientIP()` が、クライアント送信ヘッダ `cf-connecting-ip` → `x-real-ip` → `x-forwarded-for`（`.split(',')[0]` = 最左 = クライアント制御値）を**無条件に信頼**する。リクエストごとにヘッダを変えれば、ログイン/MFAのブルートフォース制限を完全回避できる。
2. **fail-open**: `src/lib/rate-limiting/rate-limiter.ts:208-218` で Redis エラー時に `allowed: true` を返す。Redisへ負荷をかければレート制限が黙って無効化される。

#### 該当箇所
- `src/lib/rate-limiting/middleware.ts:278-298`
- `src/lib/rate-limiting/rate-limiter.ts:202-218`

#### 攻撃シナリオ
攻撃者が `X-Forwarded-For: <ランダム>` を毎回付与し、`login:<ip>` バケットを毎回新規化 → パスワード総当たり。ホワイトリスト（`isWhitelisted`）も同じスプーフ可能IPでキーされるため、既知のホワイトIPを詐称すれば制限を完全回避（R-3）。

#### 改修内容

**(1) 信頼できるプロキシ段数に基づく実IP抽出**
```ts
// src/lib/rate-limiting/middleware.ts
// デプロイ先（Vercel等）が付与する検証済みヘッダを優先し、
// クライアント制御の最左値は採用しない。
function getClientIP(request: NextRequest): string {
  // 1) プラットフォーム検証済みヘッダ（例: Vercel は x-vercel-forwarded-for を付与）
  const platformIp = request.headers.get('x-vercel-forwarded-for');
  if (platformIp) return platformIp.split(',')[0].trim();

  // 2) 信頼プロキシ段数を考慮し、XFF の「右からN番目」を実クライアントとする
  const trustedProxyCount = Number(process.env.TRUSTED_PROXY_COUNT ?? '1');
  const xff = request.headers.get('x-forwarded-for');
  if (xff) {
    const parts = xff.split(',').map((s) => s.trim()).filter(Boolean);
    const idx = parts.length - trustedProxyCount;
    if (idx >= 0 && parts[idx]) return parts[idx];
  }

  // 3) Cloudflare 経由が確定している場合のみ cf-connecting-ip を信頼
  if (process.env.TRUST_CF_HEADERS === 'true') {
    const cf = request.headers.get('cf-connecting-ip');
    if (cf) return cf.trim();
  }

  return '127.0.0.1';
}
```
> デプロイ環境に応じて `x-vercel-forwarded-for` / `TRUSTED_PROXY_COUNT` / `TRUST_CF_HEADERS` を設定する。**クライアントが任意に設定できる最左XFF・`x-real-ip`・`cf-connecting-ip` を無条件採用しない**ことが要点。

**(2) ログイン/MFAは account+IP の複合キー**
IP回転に耐えるよう、認証系のレート制限キーに正規化したアカウント識別子（email等）を併用する。例: `login:<emailHash>:<ip>`。これにより単一アカウントへの総当たりはIPを変えても抑止される。

**(3) 認証系は fail-closed**
```ts
// rate-limiter.ts — 認証系タイプ（login/mfa）かつ本番では Redis エラー時に拒否
} catch (error) {
  log.error('レート制限チェックエラー:', error);
  const failClosed =
    process.env.NODE_ENV === 'production' &&
    (type === 'login' || type === 'mfa');
  if (failClosed) {
    return { allowed: false, limit, remaining: 0, resetTime: now + window };
  }
  return { allowed: true, limit, remaining: limit, resetTime: now + window };
}
```

#### 受け入れ条件
- [ ] クライアント送信の `X-Forwarded-For` 最左値・`x-real-ip`・`cf-connecting-ip`（非Cloudflare構成）でレート制限バケットを新規化**できない**。
- [ ] ログイン総当たりが、IPを変えても同一アカウントに対して閾値で遮断される。
- [ ] 本番で Redis 障害時、login/MFA エンドポイントが fail-closed（拒否）になる。

#### テスト
- `src/__tests__/security/` に追加：
  - スプーフヘッダを付与しても同一クライアント扱いになることの検証。
  - `getClientIP` の各分岐（platform / XFF段数 / CF）単体テスト。
  - Redis モックを失敗させ、login/mfa が本番設定で `allowed:false` を返すこと。

### SEC-04 🟠 MFA のサーバー側強制（ステップアップ認証）

#### 問題
`src/app/api/mfa/verify/route.ts` は `{ isValid }` を返すだけで、**セッションのAAL昇格も `mfa_verified` 相当のサーバー状態の記録も一切行わない**。`aal2` / `requireMFA` 等の強制点はコードベース全体でゼロ。結果、MFA有効ユーザーでも `/dashboard` や `/api/*` へ直接アクセスすれば素通りでき、MFAは**装飾（no-op）**になっている。一次認証情報を盗まれた攻撃者に対して無防備。

#### 該当箇所
- `src/app/api/mfa/verify/route.ts:74-77`
- `middleware.ts`（AALチェックなし）
- `src/lib/supabase/guards.ts`（`ensureClinicAccess` にMFAチェックなし）

#### 改修内容（推奨: 段階導入）

**方式B（短期・低破壊）— サーバー署名のステップアップ・クッキー**
1. `/api/mfa/verify` 成功時に、`httpOnly` `Secure` `SameSite=Lax` の**署名付きステップアップ・クッキー**を発行する。
   - ペイロード: `userId:sessionId:expiresAt`、署名は**専用シークレット**（SEC-10と連動。`SUPABASE_SERVICE_ROLE_KEY` は使わない）でHMAC。
   - TTLは短め（例: セッション有効期間または30分）。`sessionId` を含めセッションに束縛し、別セッションへ流用不可とする。
2. `middleware.ts` の保護ルート判定の直後に、**MFA有効ユーザー（`user_mfa_settings.is_enabled = true`）** に対して有効なステップアップ・クッキーが無ければ `/mfa`（チャレンジ画面）へリダイレクト。
3. `ensureClinicAccess`（API側）でも同様にMFA必須ユーザーのステップアップ未完了を 401/403 で弾く。
4. TOTP受理窓 `window` の上限を 4→1 に縮小（`mfa/verify` の総当たり面を縮小）。MFAレート制限キーを IP のみから **user単位**へ拡張（SEC-03(2)と整合）。

**方式A（中期・推奨の最終形）— Supabase ネイティブ MFA（AAL2）への移行**
- Supabase の MFA（factor enroll/challenge/verify）を採用し、保護ルート/APIで `getAuthenticatorAssuranceLevel().currentLevel === 'aal2'` を必須化。カスタムTOTP実装（`user_mfa_settings`）を段階的に置換。
- 本仕様では方式Bを先行実装し、方式Aを次期マイルストーンの方針として記載する。

#### 受け入れ条件
- [ ] MFA有効ユーザーが、TOTP/バックアップコード検証を経ずに保護ルート・保護APIへアクセスすると**拒否される**。
- [ ] `/api/mfa/verify` 成功後にのみ保護リソースへ到達できる。
- [ ] ステップアップ・クッキーは `httpOnly`/`Secure`/署名付きで、別ユーザー/別セッションに流用できない。
- [ ] TOTP `window` のデフォルト上限が 1。

#### テスト
- `src/__tests__/security/` に追加：
  - MFA有効ユーザーがステップアップ未完了で `/api/*` 保護ルートにアクセス → 401/403。
  - `/api/mfa/verify` 成功 → 有効クッキー発行 → 同セッションで保護リソース到達可。
  - 改竄/期限切れ/別セッションのクッキーが拒否されること（`timingSafeEqual` 経路）。

### SEC-05 🟡 マネージャー集計スコープの統一

#### 問題
マネージャーのスコープ解決に**2つの不整合なモデル**が併存する。
- **厳格（意図された境界）**: `resolveManagerAssignedClinicIds` / `resolveEffectiveClinicScope`（`src/lib/auth/manager-scope.ts:67,141`）→ `manager_clinic_assignments`。`clinics/accessible` や `manager/daily-reports/overview` はこれを使う。
- **広域**: `permissions.clinic_scope_ids`（JWTフックが**同一親組織配下の全兄弟クリニック**を充填）。

集計系で `clinic_id` 未指定時に**広域パス**へ落ちる：
- `src/app/api/admin/dashboard/route.ts:154-166`（`isAreaManagerRole` → `createScopedAdminContext(permissions)` → 親組織全クリニックを集計）
- `src/app/api/admin/tenants/route.ts:648-694`（manager + `include_kpi=true` で全兄弟クリニックのKPIを返す）

#### 攻撃シナリオ
クリニックBのみ割当のマネージャーが `GET /api/admin/dashboard`（`clinic_id` 省略）を呼ぶと、同一親組織配下の全クリニックA…Zの売上・患者数・スタッフ成績を取得できる。`clinics/accessible` は割当外を隠すため、これはアプリ全体が守ろうとしている権限境界の不整合。

#### 改修内容
集計系の `isAreaManagerRole`／manager 分岐で、`createScopedAdminContext(permissions).scopedClinicIds`（広域）ではなく **`resolveEffectiveClinicScope` / `resolveManagerAssignedClinicIds`（割当ベース）** を用いてスコープを解決する。`admin` ロールは従来通り親組織全体で良い。
```ts
// admin/dashboard, admin/tenants の manager 分岐の擬似コード
const scopedClinicIds =
  auth.role === 'admin'
    ? createScopedAdminContext(permissions).scopedClinicIds // 親組織全体
    : await resolveManagerAssignedClinicIds(auth.id);       // 割当のみ
```

#### 受け入れ条件
- [ ] 単一割当マネージャーが集計エンドポイントを `clinic_id` 省略で呼んでも、**割当クリニックのみ**が集計対象になる。
- [ ] `admin` ロールの挙動は不変。
- [ ] `clinics/accessible` と集計系のスコープが一致する。

#### テスト
- 既存 `src/__tests__/api/admin-dashboard-route.test.ts` / `admin-tenants-access.test.ts` / `multi-store-kpi.test.ts` を拡張し、割当外クリニックが集計に**含まれない**ことを検証。

> **要確認**: 「マネージャー＝エリアマネージャー（親組織全体が正）」という運用も理論上はあり得る。`manager_clinic_assignments` が全所で境界として使われている事実から本仕様は**割当ベースが正**と判断するが、もし運用上エリア集計が正なら、その場合は `clinics/accessible` 側を広域に揃える形で整合を取る（実装前にプロダクト確認）。

### SEC-06 🟡 `/api/chat` の認可とスキーマ検証

#### 問題
`src/app/api/chat/route.ts`（POST）で `clinic_id` 省略時、`requireClinicMatch` が `false` となりロール検査が行われないまま、`is_admin_session: !clinic_id` → `true` のセッションが作成される。任意の認証ユーザー（低権限スタッフ含む）が管理者/本部チャットセッションを作成できる。加えて `message` 等がZod未検証で、長さ無制限のペイロードがDB挿入まで到達する。

#### 該当箇所
- `src/app/api/chat/route.ts:63-102`

#### 改修内容
1. `is_admin_session` を**パラメータ不在から導出しない**。明示的にロール（`ADMIN_UI_ROLES`/`HQ_ROLES`）を検査し、特権ロールのみ clinic-less（管理者）セッションを許可する。非特権ユーザーには `clinic_id` を必須とする。
2. リクエストボディに Zod スキーマを適用：
```ts
const ChatPostSchema = z.object({
  message: z.string().min(1).max(4000),
  clinic_id: z.string().uuid().optional(),
  session_id: z.string().uuid().optional(),
});
```
3. `user_id` はサーバー導出（`result.auth.id`）のみを使用し、ボディ由来値を信用しない（mass-assignment防止）。

#### 受け入れ条件
- [ ] 非特権ユーザーが `clinic_id` 省略で POST → 400/403（管理者セッションを作成できない）。
- [ ] `message` の型・長さがスキーマで強制される。
- [ ] `is_admin_session` はロール検査の結果としてのみ true になる。

#### テスト
- `src/__tests__/api/` に chat ルートテストを追加（特権/非特権 × clinic_id有無のマトリクス、長大message拒否）。

### SEC-09 🟢 PIIテーブルの anon/authenticated 権限剥奪

#### 問題
`supabase/migrations/00000000000001_squashed_baseline.sql:5736+` が `patients`/`revenues`/`customers`/`visits`/`daily_reports`/`staff`/`profiles`/`clinics` 等に `GRANT ALL ... TO anon` / `authenticated` を付与。後続のハードニング移行（`20260413000100`）は一部テーブルのみ剥奪し、**PIIテーブルは未剥奪**。現状RLSで保護されているが、RLS誤無効化や `USING (true)` 追加が1つでもあれば匿名で全PIIが露出する。

#### 改修内容
新規マイグレーションを追加：
```sql
-- 例: supabase/migrations/<timestamp>_revoke_anon_pii_grants.sql
REVOKE ALL ON public.patients      FROM anon;
REVOKE ALL ON public.revenues      FROM anon;
REVOKE ALL ON public.customers     FROM anon;
REVOKE ALL ON public.visits        FROM anon;
REVOKE ALL ON public.daily_reports FROM anon;
REVOKE ALL ON public.staff         FROM anon;
REVOKE ALL ON public.profiles      FROM anon;
-- 必要に応じて clinics / staff_performance / PII ビューも対象
-- authenticated は必要最小（SELECT/INSERT/UPDATE/DELETE）まで縮小し RLS に依存
-- 最重要テーブルは FORCE ROW LEVEL SECURITY を検討
ALTER TABLE public.patients FORCE ROW LEVEL SECURITY;
ALTER TABLE public.revenues FORCE ROW LEVEL SECURITY;
```
- 併せて `supabase/rollbacks/` にロールバックスクリプトを用意。

#### 受け入れ条件
- [ ] `anon` ロールが PII テーブルに直接アクセスできない（権限レベルで拒否）。
- [ ] 既存のアプリ機能（認証済みユーザーの正常系）が回帰しない。

#### テスト
- `src/__tests__/rls/` / `src/__tests__/security/` の RLS テストに、anon でのPIIアクセス拒否ケースを追加。
- ローカルSupabaseでマイグレーション適用→主要E2Eがグリーンであること。

### SEC-10 🟢 HMAC署名鍵の分離

#### 問題
`src/lib/auth/password-recovery-intent.ts:15` が **`SUPABASE_SERVICE_ROLE_KEY`（最高権限のDB鍵）** をパスワード復旧インテントCookieのHMAC署名に再利用。鍵ローテーションが認証署名とDB管理で結合し、HMACオラクル/漏洩が最高権限鍵に波及する。

#### 改修内容
- 専用シークレット `RECOVERY_SIGNING_SECRET`（または既存 `ENCRYPTION_KEY`）で署名するよう変更。
```ts
function signPayload(payload: string) {
  return createHmac('sha256', assertEnv('RECOVERY_SIGNING_SECRET'))
    .update(payload)
    .digest('hex');
}
```
- `.env.*.example` / `src/lib/env.ts` に新シークレットを追加（最小長検証はSEC-11で）。
- 検証は既存の `timingSafeEqual` を維持。
- **移行配慮**: 鍵変更で既存発行済みインテント（TTL最大10分）は失効する。デプロイ時の短時間影響として許容、もしくは新旧両鍵で一時的に検証する移行窓を設ける。

#### 受け入れ条件
- [ ] 署名/検証が `SUPABASE_SERVICE_ROLE_KEY` を参照しない。
- [ ] 新シークレット未設定時は起動時に明示エラー（`assertEnv`）。

#### テスト
- 署名→検証の往復テスト、改竄/期限切れ拒否テスト（既存テストの鍵参照を更新）。

### SEC-11 🟢 環境変数検証の強化と logger の PII redaction

## 4. 実装順序と依存関係

```
SEC-01 (Next 15.5.19)  ──┬─▶ SEC-02 (CSP nonce)        ── Phase 1
                         │
                         ├─▶ SEC-03 (Rate limit IP)     ── Phase 1
                         └─▶ SEC-04 (MFA enforce)       ── Phase 1
                                  │
SEC-10 (署名鍵分離) ─────────────┘（SEC-04のクッキー署名で利用）

SEC-05 / SEC-06 / SEC-07 / SEC-08  ── Phase 2（相互独立）
SEC-09 / SEC-11 / SEC-12           ── Phase 3（相互独立）
```

- **SEC-01 は全ての前提**（特にSEC-02のnonce XSS advisory）。
- **SEC-10 を SEC-04 より先**に入れると、ステップアップ・クッキーの署名鍵を最初から専用鍵にできる。
