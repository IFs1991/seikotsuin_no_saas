This document is the implementation specification required by `AGENTS.md` for
the PR-07 forward migration. It narrows the program specification; it does not
authorize table deletion, data repair, or any linked/production application.
Because the linked project has not yet recorded the PR-06 migration, PR-07 must
not be applied there even if a PR-07 operator approval is requested in
isolation. PR-06 history parity is a prerequisite.

## 2. Objective

Remove five legacy relations from every commercial application write path while
preserving their data and the two reviewed server-side read paths. Contain the
current nullable-clinic risk fail-closed: a migration apply is refused if a
`visits` or `revenues` row has an unresolved `clinic_id`. PR-07 deliberately
does not make either column `NOT NULL`; row repair and a future schema constraint
require their own reviewed migration.

## 3. In scope

| Table | Tenant state | Runtime decision | Final service role access |
| --- | --- | --- | --- |
| `appointments` | `clinic_id NOT NULL`; superseded by `reservations` | migration-only | none |
| `visits` | nullable `clinic_id`; legacy heatmap source | reviewed read-only | `SELECT` |
| `revenues` | nullable `clinic_id`; legacy analysis/heatmap source | reviewed read-only | `SELECT` |
| `treatments` | `clinic_id NOT NULL`; no runtime reference | migration-only | none |
| `treatment_menu_records` | no `clinic_id`; tenant meaning is indirect | migration-only | none |

The quarantine contract is:

1. RLS is enabled on all five relations.
2. No RLS policy remains on any of the five relations.
3. `PUBLIC`, `anon`, and `authenticated` receive no direct relation privilege.
4. `service_role` receives only `SELECT` on `visits` and `revenues`.
5. `service_role` receives no relation privilege on the other three tables.
6. No application write to any candidate is allowed.
7. Ownership stays with `postgres`, so reviewed migrations can inspect or repair
   data after separate approval.
8. No row, table, column, constraint, or index is deleted by PR-07.

The related function boundary is also exact:

1. `public.get_hourly_visit_pattern(uuid)` remains `SECURITY INVOKER` with its
   reviewed fixed `search_path`, exact signature/execution attributes, and
   reviewed body containing both the visit-clinic predicate and same-clinic
   revenue join.
2. Only `service_role` receives `EXECUTE`.
3. `PUBLIC`, `anon`, and `authenticated` receive no `EXECUTE` privilege.

The machine-readable target is
`docs/stabilization/evidence/commercial-hardening/pr07/legacy-quarantine-matrix.csv`.

## 5. Nullable clinic remediation

PR-07 does not invent a tenant for ambiguous data and does not silently make a
legacy column `NOT NULL` merely because the current tables are empty.

The forward migration checks `visits.clinic_id` and `revenues.clinic_id` before
changing access metadata. If either null count is non-zero, it aborts before any
DDL. The repair order is:

1. derive the clinic uniquely from a trusted parent relation;
2. otherwise derive it uniquely from an audited source event;
3. otherwise obtain a documented owner decision;
4. if none applies, keep the rollout blocked.

Any repair requires a separate specification, RED fixture, forward migration,
and operator approval. Arbitrary assignment to a default clinic is prohibited.
Because both observed null counts are zero and application writes become
impossible, there is no unresolved row to repair in this PR. The generated type
contract for both columns remains nullable after PR-07.

## 6. `treatments` decision

Both treatment relations are quarantined now:

- `treatments` has no production runtime reference and belongs to the legacy
  `appointments` model rather than the `reservations` SSOT.
- `treatment_menu_records` has no production runtime reference and no direct
  tenant key, so restoring client access would create an ambiguous boundary.

Neither relation is deleted. Retention and deletion prerequisites are isolated
in `docs/stabilization/spec-commercial-legacy-deletion-candidates-v0.1.md`.

## 7. Forward migration

The migration must:

1. set short lock and statement timeouts;
2. verify the five exact relations, owners, expected nullable columns, expected
   policy set, and required roles;
3. snapshot row counts inside the transaction;
4. abort if `visits` or `revenues` contains a null `clinic_id`;
5. enable RLS on all five tables;
6. drop the nine exact stale policies;
7. revoke all relation privileges from `PUBLIC`, `anon`, `authenticated`, and
   `service_role`;
8. grant only `SELECT` on `visits` and `revenues` to `service_role`;
9. revoke function `EXECUTE` from `PUBLIC`, `anon`, `authenticated`, and
   `service_role`, then grant it back only to `service_role`;
10. verify that `get_hourly_visit_pattern(uuid)` retains its exact reviewed
    signature, execution attributes, body, `SECURITY INVOKER` mode, and fixed
    `search_path`;
11. add catalog comments describing the quarantine and no-delete decision;
12. verify ACLs, RLS, zero policies, zero nullable rows, owner preservation, and
    unchanged row counts before commit.

The migration must not update data, set/drop `NOT NULL`, rename or drop a
relation, or restore client access.

## 8. Tests and acceptance

- RED runner reproduces `COMM-LEGACY-001` before the migration and advances only
  that contract to GREEN afterward.
- pgTAP proves exact RLS/policy/ACL state, direct client denial, service-role
  write denial, and the two required server reads.
- Jest proves the paired spec/rollback/evidence, forbids data/destructive SQL,
  and runs the deterministic runtime-reference verifier.
- The runtime verifier proves that E2E seed/cleanup and the connection probe do
  not reintroduce a legacy table reference.
- Targeted Playwright proves direct client denial for `visits`/`revenues` and
  the dashboard's intentional empty heatmap state after legacy fixture writes
  are removed. The program-wide full Playwright gate remains a separate result.
- Existing PR-02 privilege and PR-03 RLS tests remain green.
- Migration replay, generated-type parity, focused/full Jest, security tests,
  lint, type-check, build, and secret scan are required before handoff.
- At least two independent read-only subagent audits are required after the
  implementation is complete.

Acceptance is met when there is no application write, no direct client access,
the reviewed reads still work, row counts are unchanged, no unresolved nullable
row exists, and deletion has not occurred.

## 9. Recovery

Restoring service-role writes or authenticated policies would reintroduce the
closed boundary. The paired rollback is therefore validation-only and
security-preserving. If a reviewed read breaks:

1. disable the affected dashboard/analysis route;
2. preserve the quarantine ACL and deny-all RLS state;
3. ship a reviewed least-privilege forward fix.

Running the recovery SQL, applying the migration to linked/staging/production,
or repairing data requires explicit operator approval.
