## 1. エグゼクティブサマリー

コードレベルのセキュリティ設計（RLS・テナント分離・認可・service role 集約）は、単独開発のSaaSとしては**例外的に高水準**。一方、エンタープライズ調達（特に医療情報を扱う事業者の調達審査）で必ず問われる **可用性(SLA/DR)・観測性・第三者証跡・医療情報ガイドライン適合**が未実装で、現状では「PoCには出せるが本契約の監査には未達」。`0.1.0-pilot` という自己認識は正確。

### 成熟度サマリー（5段階）

| 領域 | 評価 | 要約 |
|------|------|------|
| マルチテナント分離 | ★★★★☆ | RLS 148ポリシー + scoped admin。設計は堅牢だが分析RPC経路に多層防御の穴 |
| 認証・認可 | ★★★★☆ | MFA暗号化・JWT custom claims hook・RBAC正規化層あり。MFA強制は未実装 |
| アプリ層セキュリティ | ★★★☆☆ | rate-limit fail-closed・監査ログ・secret scan。CSPは fail-open |
| CI / 品質ゲート | ★★★☆☆ | gateは通るが**セキュリティテストが非ブロッキング**・E2E未実行 |
| 可用性・DR | ★☆☆☆☆ | RTO/RPO・バックアップ・HA・復旧訓練の文書なし |
| 観測性・運用 | ★★☆☆☆ | Sentryのみ。メトリクス/集中ログ/アラート/深いヘルスチェック欠落 |
| 医療コンプライアンス | ★★☆☆☆ | 監査ログ基盤はあるが3省2ガイドライン適合の証跡なし |

## 2. 強み（このまま伸ばすべき資産）

| # | 強み | 根拠 |
|---|------|------|
| S-1 | **RLSが fail-closed で機能**。`patients` 等で `get_current_role() = ANY(...) AND can_access_clinic(clinic_id)` の二重条件、書き込みは service_role 限定（`patients_insert_legacy_block`） | `supabase/migrations/00000000000001_squashed_baseline.sql:5068-5080`、148ポリシー |
| S-2 | **service role 利用の中央集約**。`createScopedAdminContext()` が scope 検証を強制し `assertClinicInScope()` を提供 | `src/lib/supabase/scoped-admin.ts` |
| S-3 | **rate limiter が本番 fail-closed**（Redis障害時に 503）。多くのSaaSがfail-openで事故る箇所 | `src/lib/rate-limiting/middleware.ts:118` |
| S-4 | **MFAシークレットをアプリ層で暗号化**（AES + `ENCRYPTION_KEY`） | `src/lib/mfa/crypto.ts` |
| S-5 | **マイグレーション+ロールバック+仕様書のセット運用**（rollbacks 30本）、secret scan のCI必須化、pen-testチェックリスト整備 | `supabase/rollbacks/`、`.github/workflows/ci.yml`、`docs/operations/PENETRATION_TEST_CHECKLIST.md` |
| S-6 | **APIルートの定型化**（Zod検証 → `processApiRequest()` → `ensureClinicAccess()` → 統一エンベロープ → `handleRouteError()`）。境界の一貫性が高い | `src/lib/api-helpers.ts`、`src/lib/supabase/guards.ts` |

> **総括**: コアの作り直しは不要。不足は「周辺の固さ」であり、医療×マルチテナントでは技術的難易度より調達上の重みが大きい。

## 3. ギャップ詳細と対策

重大度: 🔴 Critical（契約前監査で必ず指摘）／🟠 High／🟡 Medium

### 🔴 C-1. 可用性・DR設計が不在

- **所見**: `docs/operations/` に backup / disaster recovery / RTO / RPO / incident-response の文書がない。Supabaseのマネージドバックアップ任せで、復旧手順・訓練記録が存在しない。
- **根拠**: `docs/operations/`（RUNBOOK.md・BETA_OPERATIONS.md 等はあるがDR文書なし）。
- **リスク**: エンプラ調達票の「RTO/RPOは何分か」「リージョン障害時の切替」「復旧訓練の記録」に回答不能 → 失注。データ消失時に復旧不能。
- **対策**:
  1. Supabase **PITR（Point-in-Time Recovery）**を有効化し、保持期間を契約要件に合わせる。
  2. `docs/operations/DR-PLAN-v0.1.md` を作成：RTO/RPO定義、バックアップ方式、復旧手順（手順を伴うrunbook）、責任者・連絡網。
  3. **年次以上の復旧訓練**を実施し記録テンプレートを残す（監査で「訓練実績」を問われる）。
  4. バックアップの**別リージョン/別アカウント保管**で単一障害点を除去。

### 🔴 C-2. セキュリティテストがCIのマージゲートでない

- **所見**: `test-security` ジョブが `continue-on-error: true`。`security|session-management|rls` のテストが失敗してもマージが通る。
- **根拠**: `.github/workflows/ci.yml:191`。
- **リスク**: テナント分離・認可のリグレッションをCIが止められない。医療系で `clinic_id`/`role`/`user_id` を扱う以上、致命的。
- **対策**:
  1. `continue-on-error` を除去し、`test-security` を**必須ゲート化**。
  2. 不安定なテストは個別に `skip` 理由を明示し、スイート全体を非ブロッキングにしない。
  3. ブランチ保護ルールで `test-security` を required status check に登録。

### 🔴 C-3. E2E（Playwright）がCIで実行されていない

- **所見**: Windows の spawn EPERM を理由にPlaywrightがCIから除外。認可・テナント越境の統合シナリオが自動回帰されていない。
- **根拠**: `.github/workflows/ci.yml` 冒頭コメント「Playwright – blocked by spawn EPERM on Windows」。
- **リスク**: 単体テストが通っても、実ブラウザ経由の越境・権限昇格が検知されない。
- **対策**:
  1. **Linux runner（ubuntu-latest）でPlaywrightを実行**するジョブを追加（EPERMはWindows固有のため解消）。
  2. 最低限「ロール別アクセス境界」「テナント越境拒否」の主要シナリオを必須化。
  3. seed/cleanup（`npm run e2e:seed` / `e2e:cleanup`）をCIに組み込む。

### 🔴 C-4. 患者データのアプリ層暗号化・鍵管理が未文書

- **所見**: `encrypt` 利用はMFAのみ。患者の要配慮個人情報は Supabase の at-rest 暗号化に依存し、暗号化方式・鍵管理・アクセス記録が文書化されていない。
- **根拠**: `grep encrypt src/lib` の結果が `mfa/crypto.ts`・`mfa/mfa-manager.ts`・`env.ts` のみ。
- **リスク**: 医療情報の「3省2ガイドライン」「個人情報保護法（要配慮個人情報）」適合の証跡欠如。委託先（Supabase/Vercel）管理の説明責任も未整備。
- **対策**:
  1. **適合マッピング文書**を作成（暗号化:at-rest/in-transit、鍵管理:KMS/ローテーション、アクセス記録、委託先管理）。
  2. 機微カラム（診療メモ等）の**アプリ層暗号 or pgcrypto** 適用を検討。
  3. `ENCRYPTION_KEY` を `src/lib/env.ts` の**必須env**に昇格（現在は任意で、欠落時に暗号が無効化される恐れ）。

### 🟠 H-1. テナント分離の多層防御が分析RPC経路で1枚だけ

- **所見**: `manager_revenue_*` の3 RPCは `security invoker` かつ `service_role` 限定 grant。ルートが `createAdminClient()`（service_role）で呼ぶため**RLSをバイパス**し、RPC本体は `p_clinic_ids` を**無検証で信頼**する。テナント境界は**アプリ層ルートのスコープ解決1枚のみ**で、RLS/RPCの backstop が無い。
- **根拠**:
  - RPC定義: `supabase/migrations/20260611000200_manager_revenue_analysis_rpcs.sql:36`（`security invoker`）、`:373-386`（`revoke ... from authenticated; grant ... to service_role`）。
  - 呼び出し: `src/app/api/manager/revenue/analysis/route.ts:58`（`createAdminClient()`）、`:59-74`（`resolveManagerAssignedClinics` + 403チェック）。
- **評価**: **現行ルートは正しくスコープしており、現時点の実害ではない**（`authenticated` からは revoke 済みのため、ユーザーがPostgREST経由でRPCを直接叩くこともできない）。問題は **CLAUDE.md の不変条件「RLSが最後の砦」がこの経路では成立せず**、防御が単層であること。新規ルートの1つのミス（request由来の `clinic_ids` を検証せずRPCへ渡す等）や `resolveManagerAssignedClinics` のバグが、**直ちに無声のテナント越境漏洩**になる。
- **対策**:
  1. **ESLintルール**で route内の plain `createAdminClient` 直呼びを禁止し、`createScopedAdminContext()`（scope assert付き）経由を強制（現状 plain使用ファイルが約29件）。
  2. service_role-only RPCを呼ぶ箇所に、呼び出し直前の `assertClinicInScope()` を規約化。
  3. 分析RPCに対する**テナント越境の負のテスト**（別テナントのclinic_idsを渡して0件/拒否を検証）を必須スイートへ追加。
  4. 中期: RPC内に `can_access_clinic()` 相当のチェックを入れ、**RLS的backstopを二層目として復活**させる。

### 🟠 H-2. 観測性がSentry一本

- **所見**: `src/lib/monitoring/` は `sentry.ts` のみ。構造化集中ログ・メトリクス/APM・アップタイム監視・アラート閾値が欠落。
- **根拠**: `src/lib/monitoring/sentry.ts`（他なし）。
- **リスク**: SLA運用に必要な「障害に気づく仕組み」が弱い。MTTR悪化。
- **対策**: ①集中構造化ログ（リクエストID/clinic_id/role付与）②メトリクス（p95レイテンシ・エラー率・429率・RLS拒否率）③外形監視（Uptime）④主要閾値のアラート（PagerDuty/Slack）。

### 🟠 H-3. ヘルスチェックが浅い

- **所見**: `/api/health` は `clinics` を1行SELECTのみ。Redis・外部依存（Resend/Gemini）・認証の死活を含まない。
- **根拠**: `src/app/api/health/route.ts`。
- **リスク**: LB/監視からは「DBだけ生きていれば healthy」に見え、Redis障害（=rate-limit fail-closedで全503）等を検知できない。
- **対策**: `/api/health`（軽量・LB用）と `/api/health/deep`（DB+Redis+外部依存、内部監視用）を分離。各依存にタイムアウトと個別ステータスを付与。

### 🟠 H-4. MFA強制ポリシーが未実装

- **所見**: MFA基盤はあるが、ロール別の強制ロジックが `src/lib/auth` に見当たらない。
- **根拠**: `grep mfa|enforce src/lib/auth` がヒットなし。
- **リスク**: admin/clinic_admin が単一要素のままだと、管理者アカウント侵害が1要素で成立。
- **対策**: admin/clinic_admin ロールのMFAを**必須化**（未設定はMFA設定画面へ強制リダイレクト）。middlewareまたはguardで enforce。

### 🟡 M-1. CSPが fail-open

- **所見**: CSP生成失敗時に fail-open（middleware）。
- **リスク**: 攻撃・障害時にむしろCSP保護が外れる。
- **対策**: 段階適用（`CSP_ROLLOUT_PHASE`）の **full-enforce 期日**を切る。enforce後は fail-closed を検討。

### 🟡 M-2. 型安全の規律と実態の乖離

- **所見**: `tsconfig` は `strict: false`、`src/__tests__` は `tsc --noEmit` 対象外。
- **根拠**: プロジェクトメモ（tsconfig.json Quirks）。
- **リスク**: テスト・境界コードの型崩れが検知されにくい。
- **対策**: API境界（Zod）での実行時検証を維持しつつ、段階的に `strict` 有効化（`strictNullChecks` から）。

### 🟡 M-3. 負荷試験・スケール上限が未検証

- **所見**: シングルテナントDB構成の同時接続・スループット上限の実測がない（`phase4a` benchmarkは範囲限定）。
- **対策**: 想定ピーク（同時テナント数×予約スループット）での負荷試験。接続プール（`supabase/config.toml [db.pooler]`）のmode/上限を実測に基づき設定。
