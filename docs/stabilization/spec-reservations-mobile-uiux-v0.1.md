## 0. この素案の位置づけ

予約画面（`/reservations`）を**スマートフォン(<768px)で実用的に使えるUI/UX**に最適化するための、チーム実装向けの叩き台。

- 本v0.1は **既存のタイムライン・スケジューラ（PC体験）を温存したまま**、モバイル専用の表示経路を「追加」する方針。Scheduler本体の破壊的改修は対象外（§3.2）。
- **認可・テナント分離・clinic scope の不変条件は一切緩めない**。`canWriteReservationsForClinic` / `isCrossClinicReservationView` / manager read-only など既存のガードをそのまま通す（AGENTS.md / [[manager-analytics-rpc-scope]]）。
- TDDで進める（CLAUDE.md）。各フェーズは「失敗するテスト → 最小実装 → リファクタ」。
- 日付・時刻は必ず `src/lib/jst.ts` のJSTユーティリティを使う（DoD-06、UTC/JST混在禁止）。

## 1. 要約

予約画面は本システムの中核だが、現状は**デスクトップ専用の二次元タイムライングリッド**（縦=リソース / 横=時間、ピクセル絶対配置）であり、スマホでは実用に耐えない。具体的には次の3点が機能的な欠陥になっている。

1. **横スクロール地獄**: 狭い画面では「全体が見えない小窓」となり、当日の予約把握に多数のスクロール操作が必要。
2. **ドラッグ移動がタッチで動作しない**: `Scheduler.tsx` の予約移動は HTML5 ネイティブDnD（`onDragStart`/`onDrop`/`dataTransfer`）依存。**タッチデバイスでは発火せず、スマホで予約移動が実質不可能**（＝既存バグ）。
3. **空き枠タップ作成が不正確**: `e.clientX - rect.left` による時間逆算は、圧縮された横軸＋指タップでは誤った時間を拾う。

本仕様は、**モバイルでは「縮小タイムライン」ではなく別ビュー（アジェンダ＋縦タイムライン）へ切り替える**ことでこれらを解消する。リスト系ビュー（`AppointmentList`）の土台が既にあるため、これを主役に据える。

## 2. 背景・目的

### 2.1 背景

- 受付・施術者は施術の合間に片手・短時間で予約を確認/操作する。タブレット受付も併存。
- 現行 `Scheduler.tsx` は `PIXELS_PER_HOUR` / `SIDEBAR_WIDTH` のピクセル設計で、`md`未満の最適化は未実施。
- ロールごとに「スマホでやりたいこと」が大きく異なる（§5）。

### 2.2 目的

- スマホで **当日の予約把握・新規作成・時間/担当変更・確定/キャンセル** を破綻なく完遂できる。
- 施術者が「自分の今日の担当」を最短手数で確認できる。
- タッチ環境での**予約移動の機能回復**（現状バグの解消）。

### 2.3 非目的（v0.1）

- PC版タイムライン・スケジューラの体験変更（DnD等はPCで維持）。
- 公開予約画面 `/booking/[clinic_id]`（患者向け）の改修。別仕様。
- 予約のビジネスロジック（重複判定・スナップ間隔・ステータス遷移）の仕様変更。
- ネイティブアプリ化・オフライン対応・PWA push。

## 3. スコープ

### 3.1 対象（v0.1）

- `md`(768px)未満での `/reservations` 表示経路の追加・最適化。
- モバイル向けビュー: **アジェンダ（時系列リスト）** と **縦・単一リソース・タイムライン**。
- 予約詳細・新規作成・編集の**ボトムシート化**（既存モーダルのモバイル表現）。
- タッチでの予約移動・担当変更の代替手段（DnD非依存、§6.3）。
- ロール別デフォルトビューの出し分け（§5）。

### 3.2 対象外（明確に温存・禁止）

- `Scheduler.tsx` の二次元グリッド描画ロジックの削除・置換（PCで継続利用）。
- `appointments` テーブルへの書き込み（予約SSOTは `reservations`、CLAUDE.md）。
- 認可ガードの分岐追加以外の変更（read-only境界・cross-clinic境界を緩めない）。
- ピクセル逆算による時間決定をモバイルへ持ち込むこと（離散スロット化、§6.2）。

## 5. ロール別 モバイル要件

| ロール | 主目的 | デフォルトビュー(<768px) | 書き込み |
|---|---|---|---|
| セラピスト | 自分の今日の担当を順に確認 | アジェンダ（**自分フィルタ**初期ON） | 既存ガードに従う |
| 店舗スタッフ/受付 | 当日対応・枠確認・新規登録 | アジェンダ + 縦タイムライン切替 | 可（`canWriteReservations`） |
| 店舗管理者 | 全体の埋まり把握・調整 | アジェンダ + 縦タイムライン | 可 |
| エリアマネージャー | 閲覧のみ | アジェンダ（read-only、既存挙動踏襲） | 不可（既存 timeline-only を維持） |

- マネージャーの `timeline` 強制（`effectiveCurrentView`）はモバイルでも維持。モバイルでは閲覧用アジェンダを既定にするが、**read-onlyメッセージ・書き込み禁止は不変**。

## 6. UI/UX設計

### 6.1 ビュー切替（自動 + 手動）

- `md`未満では初期ビューをロール別（§5）に決定。`useMediaQuery`（共通フック新設、CSSで判定困難な分岐のみ）。
- 既存 `ViewMode` を活かしつつ、縦タイムライン用に **`mobile-timeline`** を追加（PCの `timeline` は不変）。
- URL `?view=` の互換は維持（マネージャーの `register`/`list` → `timeline` 置換ロジックを壊さない）。

### 6.2 アジェンダ（時系列リスト・主役）

- 時刻つきカードの縦リスト。1カード = 1予約。タップで詳細シート。
- カード要素: 開始-終了時刻 / **患者本名（フルネーム）** / メニュー / 担当 / ステータス色チップ（未確認・確定・キャンセル/no-show）。患者名は受付・施術での本人特定に必須のため本名を表示する（§6.8 のPII取り扱い条件付き）。
- 上部に `DaySummary`（予約数・空き・キャンセル件数）を sticky 固定。
- セラピストは「自分の担当のみ」トグルを上部に置き、初期ON。
- 空状態（当日予約なし）と読み込み（skeleton）を明示。

### 6.3 予約移動・担当変更（DnD代替）

- **PC**: 既存DnDを維持。
- **モバイル**: 予約カード/ブロックをタップ → 詳細シート →「時間を変更」「担当を変更」を**フォーム選択**で実行。
  - 時間変更は離散スロット（`SNAP_MINUTES`準拠）のセレクタ。
  - サーバ呼び出しは既存 `moveAppointment` / `updateAppointment` を再利用（重複判定 `hasTimeConflict` を含む既存ロジックを通す）。
  - 競合時は既存の `onMoveError`/`updateError` 表示を流用。

### 6.4 新規作成（離散スロット）

- ピクセル逆算を使わない。アジェンダ/縦タイムラインの**空き枠を実DOMボタン（44×44pt以上）**として描画し、タップで `AppointmentFormModal` をシート起動（`initialData` に resourceId/時刻を渡す）。

### 6.5 縦・単一リソース・タイムライン（`mobile-timeline`）

- 縦=時間（上→下）、リソースは上部の横スクロールチップ or セレクタで1つずつ切替。
- 現在時刻ライン（既存赤線ロジック）を縦版に流用。
- 段階実装の最後（リスク最大）。アジェンダで多くの用途は充足する前提。

### 6.6 詳細・フォームのボトムシート化（Sheet コンポーネント新設）

UX最適解として、中央Dialogの流用ではなく **ボトムシート専用コンポーネント `src/components/ui/sheet.tsx` を新設**する。

- 実装基盤は既存 `dialog.tsx` と同じ Radix Dialog プリミティブ。`side`（`bottom`/`right`等）バリアントを持たせ、`side="bottom"` を予約のモバイル詳細・フォームに使う（shadcn の Sheet 相当）。
- 既存 `src/components/ui/swipe-handler.tsx` を組み込み、**下スワイプで閉じる**。背景タップ・`Esc`・閉じるボタンでも閉じる。
- スナップ高さ: 詳細は内容高さ、フォームは全画面に近い高さ（下部固定の保存バー＋セーフエリア考慮）。
- アクセシビリティ: Radixのフォーカストラップ・`aria-modal`・初期フォーカスを維持。
- **PC(`md`以上)は従来通り中央Dialog**を使用（`AppointmentDetail`/`AppointmentFormModal` の表示器をブレークポイントで出し分け）。
- 新設 `sheet.tsx` は予約以外（患者・日報等）でも再利用できる汎用部品とし、§6 全体方針の「モバイル共通パターン」の正とする。

### 6.7 レイアウト/ナビ整合

- 予約ページ独自の `h-[calc(100vh-64px)]` は、モバイルでボトムナビ高さ＋セーフエリア分の下部余白（`pb-safe-area-bottom`）を確保し、最下部予約がナビに隠れないこと。
- `ControlBar` はモバイルで「日付ナビ + ビュー切替」だけ残し、密度切替等は「…」メニューへ。
- 色・余白はデザイントークン経由（`bg-background`/`text-muted-foreground` 等）。ベタ書き色禁止（[[既存UI02テスト]] `UI02-pilot-hardcoded-colors` の方針に整合）。

### 6.8 患者本名表示時のPII取り扱い（条件）

患者本名（フルネーム）を表示する判断（Q4）に伴い、以下を遵守する。

- **表示は認証済み業務画面 `(app)` 内、かつ clinic scope 内のデータに限る**。RLS/認可ガードを通った予約のみ表示（手書きのスコープ判定をしない）。
- **本名はUI表示限定**。ログ・監査記録・Sentry・LINE等の外部チャネルには本名を出さない（既存のPII最小化方針を維持。HQ分析LINE仕様の §3.2 と整合）。
- スクリーンショット/画面共有の運用注意は運用ドキュメント（`docs/operations/`）側に記載（受付端末の物理的な覗き見対策は運用責任）。
- 既存の患者名取得経路（予約に紐づく患者情報）を使い、**本名表示のために新たなデータ取得・権限拡大をしない**。

## 8. 受け入れ基準（Acceptance Criteria）

- AC-1: 375px幅で `/reservations` を開くと、横スクロールなしに当日の全予約を縦スクロールで把握できる。
- AC-2: セラピストでログインすると初期表示が「自分の担当のみ」のアジェンダになる。
- AC-3: スマホで予約の時間変更・担当変更ができ、サーバに反映される（**現行のDnD不能状態が解消**）。
- AC-4: 空き枠タップで作成フォームが起動し、選んだ時間・リソースが初期値に入る。意図しない時間にならない。
- AC-5: 全インタラクティブ要素のタップ領域が44×44pt以上。
- AC-6: ボトムナビ・セーフエリアに最下部の予約/保存ボタンが隠れない。
- AC-7: マネージャー/他院閲覧時の read-only・cross-clinic 制約がモバイルでも維持される（書き込み不可・メッセージ表示）。
- AC-8: ダークモードがモバイル予約画面全体で破綻しない（ベタ書き色なし）。
- AC-9: 日時表示が全てJST（`src/lib/jst.ts`）で一貫する。

## 9. テスト要件（TDD）

- **単体/コンポーネント**: ロール別デフォルトビュー決定関数（`getDefaultReservationView({role, isMobile})` 等の純関数を切り出して網羅）。`*.test.tsx`（jsdom）。
- **認可回帰（必須）**: `clinic_id`/`role` に触れるため、read-only・cross-clinic・manager 制約のテストを追加（CLAUDE.md セキュリティ不変条件）。
- **時間変更/作成**: 既存 `hasTimeConflict`/`moveAppointment` を通す経路の確認（重複時に弾く）。
- **E2E(Playwright)**: モバイルviewport（例 375×812）で「アジェンダ表示→予約タップ→時間変更→反映」「空き枠→作成」。`src/__tests__/e2e-playwright/`。`PLAYWRIGHT_BASE_URL`・seed前提。
- 既存CI必須ゲート（`test:pr05:focused`）を壊さないこと。
- **壊れた実装に合わせてテストを変えない**（CLAUDE.md）。DnDのタッチ不能はバグとして直す側。

## 10. 非機能・アクセシビリティ

- タップ領域44pt（既存 `touch-target-comfortable` を標準化）。
- フォーカスリング・`aria-selected`/`role` を新規要素にも付与（既存ボトムナビに倣う）。
- 本文16px以上・コントラストAA（高齢患者対応の受付運用も想定）。
- パフォーマンス: アジェンダは仮想化不要規模を想定（1日件数）。大量時は将来検討（§12-Q5）。

## 11. リスクと緩和

| リスク | 緩和策 |
|---|---|
| 中核機能の回帰 | Scheduler本体を変更せずモバイル経路を追加。フェーズ分割・各PRに認可テスト |

| URL/ビュー互換崩れ | `?view=` 既存ロジック（manager置換）を壊さないテストを先に固定 |
