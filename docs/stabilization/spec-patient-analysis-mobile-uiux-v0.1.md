## 0. この素案の位置づけ

患者分析画面（`/patients` の分析タブ）を**スマートフォン(<768px)で実用的に使えるUI/UX**に最適化するための、チーム実装向けの叩き台。患者一覧（`/patients/list`）は本仕様では補助的に触れ、主対象は分析ビュー。

- 本v0.1は **既存の分析ロジック・集計フック（`usePatientAnalysis` / `useManagerPatientAnalysis`）を温存したまま**、表示・チャート・フィルタのモバイル体験を最適化する方針。スコア算出・セグメント・転換率の計算は変更しない（§3.2）。
- **認可・テナント分離・clinic scope の不変条件は一切緩めない**。`isManagerRole` による分岐（`ManagerPatientAnalysisContent` ↔ `ClinicPatientAnalysisPage`）、マネージャーの担当院スコープ（`useManagerPatientAnalysis` のサーバ側スコープ）はそのまま維持（AGENTS.md / [[manager-analytics-rpc-scope]]）。
- 本画面は**患者本名（フルネーム）を多数表示する**（LTVランキング・離脱リスク・フォローアップ）。[[予約モバイル仕様 §6.8]]（`spec-reservations-mobile-uiux-v0.1.md`）と同じPII取り扱い条件を適用する（§6.6）。
- 共通モバイル部品は **予約仕様で新設する `src/components/ui/sheet.tsx` を再利用**する（重複実装しない）。
- TDDで進める（CLAUDE.md）。各フェーズは「失敗するテスト → 最小実装 → リファクタ」。
- 日付・時刻は必ず `src/lib/jst.ts` のJSTユーティリティを使う（DoD-06）。**現状 `getDateRangeDays` は `new Date('...T00:00:00.000Z')` でUTC前提のため、JST整合を確認・是正**（§6.5）。

## 1. 要約

患者分析は**読み取り専用の集計ダッシュボード**で、KPIカード・チャート（Recharts）・多列の比較テーブル・患者リストで構成される。現状はデスクトップ前提で、スマホでは次が欠陥になる。

1. **超ワイドな比較テーブル**: マネージャーの「担当院別分析」は**9列テーブル**（`overflow-x-auto` ＋ `whitespace-nowrap`）。スマホでは延々と横スクロールが必要で、院間比較が成立しない。
2. **フィルタバーの密度**: `lg:grid-cols-[minmax(160px,1fr)_...]` の横並び前提。狭幅で縦積みにはなるが、対象/期間/日付/操作ボタンの優先度設計がなく窮屈。
3. **チャートのモバイル可読性**: `ManagerChartsSection`（Recharts、`ssr:false` dynamic）が狭幅で潰れやすい。凡例・軸ラベルの省略方針が未定義。
4. **ベタ書き色・タブ**: タブが `bg-gray-200 text-gray-700`、各種 `text-gray-500` / `bg-blue-600` / `bg-blue-50 text-blue-800` 等のハードコード。ダークモードで破綻（[[既存UI02テスト]] の方針に反する）。
5. **PIIの露出設計が未整理**: 患者本名がランキング/リスク/フォローに並ぶ。表示は必要だが、外部チャネル流出防止と画面上の取り扱いを明示する必要がある。

本仕様は、**比較テーブルを院別カードへ置換**、**フィルタを「主操作のみ表示＋詳細は折りたたみ」**、**チャートはモバイル向け簡素表示**、**トークン統一とPII条件の明文化**で、スマホでの分析閲覧を成立させる。

## 2. 背景・目的

### 2.1 背景

- 店舗管理者/マネージャーは移動中・店舗間で、担当院の患者動向（新患再来率・離脱リスク・LTV・フォロー候補）をスマホで素早く確認したい。
- 現行 `page.tsx` は `isManagerRole` で **マネージャー（複数院比較）** と **単院（`ClinicPatientAnalysisPage`）** に分岐。チャートは `dynamic`/`ssr:false`。
- 単院ビューと複数院ビューでレイアウト構成が異なる（単院: フロー/サマリー/リスク/セグメント/フォロー、複数院: KPI/フィルタ/院別テーブル/選択院詳細）。

### 2.2 目的

- スマホで **担当院の患者KPI・院別比較・リスク/フォロー候補**を横スクロールなしで把握。
- チャートがモバイルでも判読できる（潰れない・凡例が読める）。
- 患者本名表示のPII条件を明文化し、外部流出を防ぐ。
- ダークモード/ライトモードで破綻しない（トークン統一）。

### 2.3 非目的（v0.1）

- 分析ロジック（スコア・転換率・セグメント・LTV・リスク判定）の算出変更。
- 集計フック/API（`usePatientAnalysis` / `useManagerPatientAnalysis` / `manager-patient-analysis`）のレスポンス仕様変更。
- マネージャー/単院の分岐条件、担当院スコープの変更。
- 「連絡する」等のアクション機能の新規実装（既存の表示・導線維持）。
- ネイティブアプリ化・オフライン対応・PWA push。

## 3. スコープ

### 3.1 対象（v0.1）

- `md`(768px)未満での `/patients`（分析ビュー：マネージャー/単院とも）の表示最適化。
- 多列テーブル（担当院別分析）の**カード化**。
- フィルタバー（対象/期間/任意期間/適用・リセット）のモバイル再構成。
- チャート（`ManagerChartsSection`）のモバイル表示方針。
- タブ・各種ハードコード色のデザイントークン化（対象画面内）。
- 患者本名表示のPII条件の明文化（§6.6）。

### 3.2 対象外（明確に温存・禁止）

- 分析の算出ロジック（フック/サーバRPC）の変更。
- マネージャー担当院スコープの解決（アプリ層スコープが唯一のテナント保証である点を踏まえ、表示最適化でスコープ判定を**手書きで再実装しない**。既存フックの結果のみ描画）。
- `isManagerRole` 分岐条件の変更。
- 患者一覧（`/patients/list`）の本格改修（タブ整合・色トークン化など軽微な整合のみ）。

## 6. UI/UX設計

### 6.1 レイアウト基本方針

- 既存 `PAGE_CLASS`（`min-h-screen bg-background p-4 sm:p-6`）を踏襲しつつ、最下部にボトムナビ高さ＋セーフエリア分の余白を確保。
- 色・余白はデザイントークン経由（`bg-card`/`text-foreground`/`text-muted-foreground`/`border-border`）。**タブの `bg-gray-200 text-gray-700`、`text-gray-500`、`bg-blue-600`、`bg-blue-50 text-blue-800`、リスクバッジの `bg-red-100` 等を当画面からトークン化**。
  - リスク/ステータスの意味色は、固定HEXでなくセマンティックトークン（warning/danger/success系）に寄せ、ダークモードでもコントラストを確保。

### 6.2 KPIサマリー（カード）

- 単院 `SummaryCards`・マネージャー `ManagerSummaryCards` とも、モバイルは **1〜2列グリッド**（`grid-cols-1`〜`grid-cols-2`）、`sm/lg` で3列へ拡張（既存の `sm:grid-cols-2 lg:grid-cols-3` を踏襲）。
- 数値を主役（大）、ラベルを `text-muted-foreground`。10項目あるマネージャーKPIは重要項目（来院患者数/新患再来率/総売上/離脱リスク高）を上位に並べる並び順を検討（§12-Q1）。

### 6.3 担当院別分析（9列テーブル → 院別カード）

最重要のモバイル課題。

- **モバイルでは `ManagerClinicComparison` のテーブルをやめ、院ごとのカード**にする（1カード=1院）。
  - カード見出し: 院名。主指標: 来院患者数 / 総売上（大）。
  - 副指標: 新患/再来/新患再来率/来院数/患者単価/離脱リスク高 を**ラベル付き2列グリッド**で。
  - 「離脱リスク高」は注意色チップで強調。
- `md`以上は**従来テーブルを維持**（切替は `useMediaQuery` 一元化、`window.innerWidth` 直読みは避ける／[[responsive-table課題]]）。
- 院数が多い場合のために、カードは院名で並べ、必要なら指標でのソート（任意・§12-Q2）。

### 6.4 フィルタバー（モバイル再構成）

- モバイルは **「対象」「期間」だけを常時表示**し、適用/リセットは下部に横並び（44pt以上）。
- 任意期間（開始日/終了日）は **period=custom のときのみ展開**（不要時は隠す）。日付は端末ネイティブ `type="date"` ピッカー。
- バリデーション（`validateManagerPatientFilters`：必須・前後関係・最大3年）はそのまま流用し、エラーは入力直下にトークン色で表示。
- 担当院セレクタ（`ManagerClinicSelector`）はフィルタ上部に独立配置（既存挙動維持）。

### 6.5 チャート（Recharts）モバイル方針

- `ManagerChartsSection` は `dynamic`/`ssr:false` を維持（バンドル/SSR配慮）。
- モバイルでは **`ResponsiveContainer` で幅100%・高さ固定（例: 220–260px）**、凡例は下部、軸ラベルは間引き（X軸の日付は省略表記）。
- 凡例が多い場合は横スクロールではなく折返し。タップでツールチップが指で読める位置に出ること。
- データ点が多い系列はモバイルで間引き表示も検討（描画は表示層のみ、集計データは非改変）。

### 6.6 患者本名表示時のPII取り扱い（条件）

本画面は患者本名（LTVランキング・離脱リスク・フォロー候補）を表示するため、[[予約モバイル仕様 §6.8]] と同一条件を遵守する。

- **表示は認証済み業務画面 `(app)` 内、かつ clinic scope 内のデータに限る**（既存フックがRLS/サーバスコープを通した結果のみ描画。スコープ判定を手書きで再実装しない）。
- **本名はUI表示限定**。ログ・監査記録・Sentry・LINE等の外部チャネルには本名を出さない（既存PII最小化方針／HQ分析LINE仕様 §3.2 と整合）。
- 本名・スコア・リスク等の表示のために新たなデータ取得経路・権限拡大をしない。
- スクリーンショット/画面共有・覗き見対策は運用ドキュメント（`docs/operations/`）側の責務として明記。

### 6.7 患者一覧（`/patients/list`）との整合（軽微）

- 分析↔一覧のタブ（`PatientTabs`）の色をトークン化し、両画面で一致させる。
- マネージャーは一覧非対応（`ManagerAccessClosedMessage`）の既存挙動を維持。一覧本体の改修は本仕様の対象外。

## 8. 受け入れ基準（Acceptance Criteria）

- AC-1: 375px幅で `/patients`（マネージャー/単院とも）を開くと、KPI・院別比較・リスク/フォローを横スクロールなしに縦スクロールで把握できる。
- AC-2: 担当院別分析がモバイルで院別カードになり、9列テーブル由来の横スクロールが発生しない。
- AC-3: フィルタは「対象/期間/適用/リセット」が指で操作でき、任意期間は custom 選択時のみ日付欄が現れる。バリデーションが従来通り機能する。
- AC-4: チャートがモバイルで潰れず、凡例・ツールチップが判読できる。
- AC-5: 全インタラクティブ要素のタップ領域が44×44pt以上。
- AC-6: ダークモードで当該画面全体が破綻しない（ベタ書き色なし。リスク/ステータス色もトークン）。
- AC-7: マネージャー担当院スコープが維持され、スコープ外の院・患者が表示されない（表示最適化でスコープが緩まない）。
- AC-8: 患者本名が外部チャネル（ログ/Sentry/LINE等）に出力されない。
- AC-9: 分析数値（KPI・転換率・LTV・リスクスコア）が最適化前後で一致する（ロジック非変更の担保）。

## 9. テスト要件（TDD）

- **単体/コンポーネント**: 院別テーブル↔カードの切替判定（`useMediaQuery` 純関数）、フィルタの custom 展開条件、リスク/ステータス色のトークンマッピング。`*.test.tsx`（jsdom）。
- **認可回帰（必須）**: `clinic_id`/`role`/スコープに触れる表示のため、マネージャー担当院スコープ（スコープ外非表示）・`isManagerRole` 分岐・一覧非対応の回帰テストを追加（CLAUDE.md セキュリティ不変条件、[[manager-analytics-rpc-scope]]）。
- **PII（必須）**: 本名がログ/Sentry/外部送信に渡らないこと（該当があれば）の確認。UI表示限定の担保。
- **フィルタ**: `validateManagerPatientFilters`（必須・前後関係・最大3年）の回帰、custom 展開/非展開の遷移。
- **日付**: `getDateRangeDays` のJST整合（境界日）テスト。
- **E2E(Playwright)**: モバイルviewport（375×812）で「KPI把握→院別カード確認→フィルタ変更→反映」「単院ビューのリスク/フォロー確認」。`PLAYWRIGHT_BASE_URL`・seed前提。
- 既存CI必須ゲート（`test:pr05:focused`）を壊さないこと。
- **壊れた実装に合わせてテストを変えない**（CLAUDE.md）。

## 10. 非機能・アクセシビリティ

- タップ領域44pt（`touch-target-comfortable` を標準化）。
- チャートは色のみに依存しない（凡例ラベル・パターン併用）。コントラストAA。
- 本文16px以上。数値の桁区切り（`toLocaleString`）維持。
- 読み込み/空/エラー状態（既存 `PageMessage`/`aria-live`）をモバイルでも維持。
- パフォーマンス: 院数・患者リスト件数は担当院規模を想定。大量時の仮想化は将来検討（§12-Q3）。

## 11. リスクと緩和

| リスク | 緩和策 |
|---|---|
| スコープ判定の巻き込み（テナント漏れ） | スコープは既存フック結果のみ描画。手書き再実装をしない／スコープ回帰テスト |
| 分析ロジックの数値差異 | フック/RPCは非変更。カード化・チャート調整は表示層に限定 |
| PII露出（外部チャネル） | 本名はUI表示限定。ログ/Sentry/LINEに出さない（§6.6・PIIテスト） |
| SSR/hydration不整合 | `window.innerWidth` 直読みを避け、CSS優先 or `useMediaQuery` 一元化 |
| チャートのモバイル劣化 | 高さ固定・凡例下部・軸間引きの方針を先に固定し、viewport別に確認 |
