# クローズドベータ運用メモ

- 作成日: 2026-03-28
- 目的:
  - クローズドベータ時点で必要な CS / 保守運用の最小骨格を日本語で残す
  - 本格運用を作り込む前に、少人数ベータで混乱しない状態を作る

## 1. このメモの前提

- 今回の段階は一般公開ではなく、クローズドベータである
- 目的は売上最大化ではなく、利用実態と運用上の詰まりを観測すること
- そのため、提供機能は限定し、問い合わせ窓口と一次対応を明確にする

## 2. クローズドベータで提供するもの

- 院内スタッフ / 管理者向けの基本運用導線
  - 認証
  - 権限制御
  - 予約運用
  - ブロック管理
  - スタッフシフト管理
  - クリニック単位のアクセス制御
- 管理系の基本設定
- ダッシュボード系の主要導線
- 監視・セキュリティ基盤
  - ヘルスチェック
  - 監査ログ
  - CSP
  - Sentry
  - レート制限

## 3. クローズドベータで提供しないもの

- 患者向け完成予約画面
- 課金機能
- 一般公開を前提とした広いサポート範囲

### 3-補足. 実装済みだが暫定扱いの導線

- `notifications`
  - 実装は存在するが、クローズドベータでは一次運用補助として扱う
  - 本格的な通知運用の提供範囲は別途整理する
- `system/status`
  - 実装は存在するが、`system_events` テーブル未整備のため状態判定は暫定
  - クローズドベータでは参考指標として扱い、SLO 判定の正本には使わない
- `/admin/security-*`
  - 関連 API やテスト対象は存在する
  - ただし pilot mode では `middleware.ts` `PILOT_BLOCKED_ROUTE_PREFIXES` により UI 導線を `/dashboard` へリダイレクトする

## 4. ベータ参加対象

- 少数の院に限定する
- 参加者の役割を事前に固定する
  - admin
  - clinic_admin
  - staff / therapist
- 参加院ごとに連絡責任者を1名決める

## 5. 問い合わせ窓口

- 問い合わせ窓口は一本化する
- 連絡手段はベータ期間中に増やしすぎない
- まずは以下を固定する
  - 受付チャネル
  - 受付時間
  - 緊急連絡の条件

## 6. 一次対応の基本方針

- 不具合報告を受けたら、最初に以下を確認する
  - 発生日時
  - 利用者ロール
  - 対象クリニック
  - 実行した操作
  - 画面URL
  - エラーメッセージの有無
- 再現条件が不明な場合は、操作ログとスクリーンショットを優先して集める
- すぐに直せない場合は、回避策を先に返す

## 7. 障害の分類

- P1: ベータ継続不可
  - ログイン不能
  - 予約登録不能
  - 主要導線の 500 エラー継続
  - テナント境界に関わる異常
- P2: 一部業務に影響
  - 特定画面のみ異常
  - 一部ロールのみ異常
  - 一時回避策あり
- P3: 軽微
  - 文言ずれ
  - 表示崩れ
  - 再読み込みで解消

## 8. データ修正の扱い

- データの手修正は運用者判断で行わない
- 修正依頼は必ず記録を残す
- 修正時に残す項目
  - 対象テーブル
  - 対象 clinic_id
  - 変更前 / 変更後
  - 実施者
  - 実施理由

## 9. 緊急停止の条件

以下のいずれかが発生した場合は、ベータ継続ではなく一時停止を優先する。

- テナント分離の破綻が疑われる
- 誤った患者データ参照 / 更新が起きる
- ログイン不能が継続する
- 主要予約導線が継続的に使えない
- データ欠損や重複登録が多発する

## 10. ベータ期間中に観測するもの

- クリニックごとの利用頻度
- ロールごとのログイン頻度
- 予約登録件数
- 利用された導線
- 問い合わせ件数
- 回避策が必要だった障害の件数

## 10-補足. 2026-03-30 時点の既知制約

- `csp-security-migration.test.ts`
  - migration SQL 未作成。spec と rollback plan が前提
- Playwright 残課題
  - 12 fail が残っている
  - 文書上の「route 未実装」で一括りにせず、`security-monitor` 系は route 実装済み前提で契約ずれも疑う
- `system/status`
  - route は実装済みだが `systemStatus` は暫定的に `operational` 固定
- 手元環境差分
  - `3000` / `3001` 系の差分が残るため、問い合わせ時は再現ポートも記録する

## 11. 今後の判断材料

以下が見えてきたら、次フェーズを判断する。

- 継続利用したい院があるか
- どの機能が実際に使われるか
- どこで運用が詰まるか
- 課金対象になり得る価値がどこにあるか

## 11-補足. 一次対応の責任

- Program Lead
  - ベータ継続可否と対外判断を持つ
- Release Captain
  - 既知不具合の切り分けと修正順管理を持つ
- Evidence Owner
  - 発生記録、再現条件、回避策、DoD との差分記録を持つ

## 12. 補足

- 今回のメモは本格 CS 体制の設計ではない
- 目的は、クローズドベータで混乱しないための最小運用メモを残すこと
- 本格的な保守運用手順、SLA、課金運用、公開向けサポート設計は PMF の兆しが見えてから別途整備する
