## 0. この素案の位置づけ

日報画面（`/daily-reports`、入力 `/daily-reports/input`、編集 `/daily-reports/edit/[id]`）を**スマートフォン(<768px)で実用的に使えるUI/UX**に最適化するための、チーム実装向けの叩き台。

- 本v0.1は **既存の集計ロジック・課金/保険コンテキスト（`revenue-context` / `customer-insurance-coverage`）を温存したまま**、モバイル表示・入力体験を「最適化」する方針。料金計算・スナップショット確定ロジックの仕様変更はしない（§3.2）。
- **認可・テナント分離・clinic scope の不変条件は一切緩めない**。`role==='manager'` の閲覧専用ビュー分岐（`ManagerDailyReportsView`）と標準ビュー（`StandardDailyReportsView`）の境界はそのまま維持（AGENTS.md / [[manager-analytics-rpc-scope]]）。
- 患者本名（フルネーム）を扱うため、[[予約モバイル仕様 §6.8]]（`spec-reservations-mobile-uiux-v0.1.md`）と同じPII取り扱い条件を適用する（§6.7）。
- ボトムシート等の共通モバイル部品は **予約仕様で新設する `src/components/ui/sheet.tsx` を再利用**する（重複実装しない）。
- TDDで進める（CLAUDE.md）。各フェーズは「失敗するテスト → 最小実装 → リファクタ」。
- 日付・時刻は必ず `src/lib/jst.ts` のJSTユーティリティを使う（DoD-06、UTC/JST混在禁止）。**現状 `page.tsx` の `formatLocalDate`/`addLocalDays` はローカルタイム依存のため、JST化を是正対象に含める**（§6.6）。

## 1. 要約

日報画面は「集計の閲覧」と「日報の入力」の2系統からなる。現状いずれもデスクトップ前提で、スマホでは次が機能的・体験的な欠陥になっている。

1. **横スクロールするワイドテーブル**: マネージャービューの「売上推移」は `min-w-[720px]` の6列テーブル＋`overflow-x-auto`。スマホでは横スクロール必須で全体把握が困難。
2. **入力フォームがモバイル非最適**: `/daily-reports/input` は患者ごとの明細（メニュー・保険/自費・負担割合・金額）を1画面に積む複雑フォーム。狭幅でラベル・入力が窮屈になり、片手入力に耐えない。
3. **ダークモード破綻・ベタ書き色**: `bg-white` / `text-gray-500` / `text-gray-900` / `bg-blue-600` など多数のハードコード色。トークン未使用でダークモードが破綻（[[既存UI02テスト]] `UI02-pilot-hardcoded-colors` の方針に反する）。
4. **集計カードのグリッド密度**: KPIサマリーが `grid-cols-2 ... lg:grid-cols-6` で、モバイルでは2列固定のまま情報過多。

本仕様は、**閲覧系はカード/縦リスト化（ワイドテーブルの置換）**、**入力系はセクション分割＋明細のカード化＋固定保存バー**でモバイル完遂を実現する。

## 2. 背景・目的

### 2.1 背景

- 施術者・受付は施術後すぐ、片手・短時間で当日の施術記録（日報明細）を入力したい。
- 店舗管理者/マネージャーは外出先や店舗間移動中に、当日/直近の売上・患者数・未提出状況をスマホで確認したい。
- 現行 `page.tsx` は標準ビュー（`StandardDailyReportsView`）とマネージャービュー（`ManagerDailyReportsView`）に分岐。入力は `/daily-reports/input` の独立フォーム。

### 2.2 目的

- スマホで **当日サマリーの即時把握**（売上・患者数・未提出/要確認）を横スクロールなしで完遂。
- スマホで **日報明細の入力・編集** を破綻なく完遂（患者明細の追加/削除/金額確認）。
- ダークモード/ライトモードいずれでも破綻しない（トークン統一）。

### 2.3 非目的（v0.1）

- 料金計算・保険コンテキスト解決・スナップショット確定（`revenue-context` / `customer-insurance-coverage`）のロジック変更。
- 集計API（`useDailyReportsQuery` / `managerDailyReports.getOverview`）のレスポンス仕様変更。
- マネージャー閲覧専用境界の緩和（書き込み導線をマネージャーに出さない）。
- 月別トレンド等のチャート新規追加（既存表示の最適化に限る）。
- ネイティブアプリ化・オフライン入力・PWA push。

## 3. スコープ

### 3.1 対象（v0.1）

- `md`(768px)未満での `/daily-reports`（一覧/サマリー）の表示最適化。
- `/daily-reports/input` および `/daily-reports/edit/[id]` の入力フォームのモバイル最適化。
- ワイドテーブル（売上推移・日報一覧）の**カード/縦リスト化**。
- ハードコード色のデザイントークン化（対象画面内）。
- `formatLocalDate`/`addLocalDays` の **JSTユーティリティ化**（`src/lib/jst.ts`）。

### 3.2 対象外（明確に温存・禁止）

- 料金・保険・課金コンテキストの計算/確定ロジック（明細の金額・負担割合の導出は不変）。
- 集計データの算出（API/RPC側）の変更。
- マネージャー/標準ビューの分岐条件（`effectiveProfile.role === 'manager'`）の変更。
- `appointments` 等、予約SSOTに関わる書き込み（日報は `reservations`/`daily_reports` 系の既存導線のみ）。

| ロール | 主目的 | モバイルでの主導線 | 書き込み |
|---|---|---|---|
| セラピスト | 当日の施術記録を素早く入力 | 入力フォーム（明細カード）優先 | 可（標準ビュー導線） |

## 6. UI/UX設計

### 6.1 レイアウト基本方針

- `container mx-auto px-4` はモバイルで左右 `px-4`、下部にボトムナビ高さ＋セーフエリア分の余白（`pb-safe-area-bottom` 相当）を確保し、最下部要素がナビに隠れないこと。
- 色・余白はデザイントークン経由（`bg-background`/`bg-card`/`text-foreground`/`text-muted-foreground`/`border-border`）。**`bg-white`/`text-gray-*`/`bg-blue-600` 等のベタ書き色を当画面から排除**。

### 6.2 KPIサマリー（カード）

- マネージャーKPI（累計売上/平均売上/患者数/客単価/未提出日/要確認日）は、モバイルで **2列グリッド**（`grid-cols-2`）に統一、`md`以上で3〜6列へ拡張。
- `MetricBox` の数値は本文より大きく（可読性）、ラベルは `text-muted-foreground`。
- 「未提出日」「要確認日」は注意色（トークンの警告系。ベタ書き赤禁止）でアクセント。

### 6.3 売上推移（ワイドテーブル → カード/縦リスト）

- **モバイルではテーブルをやめ、日付ごとのカード**にする（1カード=1日）。
  - 主表示: 日付 / 総売上（大）。副表示: 保険・自費・患者数・客単価を2列の小項目で。
- `md`以上は**従来テーブルを維持**（`responsive-table.tsx` の card↔table 切替方針に倣う。ただし `window.innerWidth` 直読みのhydrationリスクを避け、CSS優先 or `useMediaQuery` に一元化）。
- 横スクロール（`overflow-x-auto`＋`min-w-[720px]`）はモバイル経路から除去。

### 6.4 日報一覧 / ステータス一覧

- 標準ビューの一覧は概ね縦リストで良好。**タップ領域を44×44pt以上**にし、「編集」ボタンの誤タップを防ぐ。
- マネージャーのステータス一覧は、日付＋ステータスチップ＋（患者数/売上）の縦カード。ステータスは色チップ（提出済み/未提出/確認済み/要確認）でトークン色を使用。

### 6.5 入力フォーム（`/daily-reports/input`・`/edit/[id]`）

最優先のモバイル課題。複雑な患者明細フォームを段階的に最適化する。

- **セクション分割**: 「日付・基本情報」「患者明細（リスト）」「合計・保存」を縦に積む。各セクションは `Card` で区切る。
- **明細のカード化**: 患者ごとの明細（患者名/メニュー/保険・自費/負担割合/金額）を**1明細=1カード**で表示。狭幅では項目を縦積み、ラベルは上配置。
- **明細の追加/削除**: 「明細を追加」ボタンは下部固定エリア付近に配置。削除は各カードのアイコン（44pt以上、誤操作防止に確認）。
- **金額・負担割合は表示のみ温存**: 既存ロジック（`deriveLegacyBillingType` / `deriveRevenueContextCodeFromBillingType` / 保険負担率）で算出した値の表示・選択肢はそのまま。計算は変更しない。
- **固定保存バー**: 画面下部に「保存」を sticky 配置（セーフエリア考慮）。スクロール中も保存に到達できる。
- **数値入力**: 金額・回数等は `inputmode="numeric"`、日付は端末ネイティブの日付ピッカー（`type="date"`）を活用。
- **大規模フォームの分割（任意・M3）**: 1明細の編集を [[共通Sheetコンポーネント]]（`src/components/ui/sheet.tsx`、`side="bottom"`）に切り出し、一覧画面の見通しを確保する案を検討（§12-Q4）。

### 6.6 日付処理のJST化

- `page.tsx` の `formatLocalDate`/`addLocalDays`（ローカルタイム依存）を `src/lib/jst.ts` のユーティリティ（`toJSTDateString` 等、入力フォームで既に使用）に置換。
- マネージャー期間レンジ（today/last7days/thisMonth）の境界をJST基準で算出し、UTC/JSTズレによる「日付が1日ずれる」事故を防ぐ（DoD-06）。

### 6.7 患者本名表示時のPII取り扱い（条件）

入力フォーム明細に患者本名（`patientName`）を表示するため、[[予約モバイル仕様 §6.8]] と同一条件を遵守する。

- **表示は認証済み業務画面 `(app)` 内、かつ clinic scope 内のデータに限る**（RLS/認可を通った日報・明細のみ）。
- **本名はUI表示限定**。ログ・監査記録・Sentry・LINE等の外部チャネルには本名を出さない（既存PII最小化方針／HQ分析LINE仕様 §3.2 と整合）。
- 本名表示・入力のために新たなデータ取得経路・権限拡大をしない。

## 8. 受け入れ基準（Acceptance Criteria）

- AC-1: 375px幅で `/daily-reports`（マネージャー/標準とも）を開くと、横スクロールなしに当日/直近の集計を縦スクロールで把握できる。
- AC-2: 売上推移がモバイルでカード表示になり、`min-w-[720px]` 由来の横スクロールが発生しない。
- AC-3: 375px幅で `/daily-reports/input` の患者明細を追加・編集・削除でき、保存ボタンに常時到達できる（固定保存バー）。
- AC-4: 全インタラクティブ要素のタップ領域が44×44pt以上。
- AC-5: ダークモードで当該画面全体が破綻しない（ベタ書き色なし）。
- AC-6: 日付表示・期間レンジが全てJST（`src/lib/jst.ts`）で一貫し、UTC/JSTズレが起きない。
- AC-7: マネージャーは閲覧専用が維持され、入力/編集導線が表示されない。
- AC-8: 料金・保険・負担割合の算出結果が最適化前後で一致する（ロジック非変更の担保）。

## 9. テスト要件（TDD）

- **単体/コンポーネント**: 売上推移のカード/テーブル切替判定（`useMediaQuery` ベースの純関数）、ステータスチップのマッピング。`*.test.tsx`（jsdom）。
- **日付（必須）**: `formatLocalDate`/`addLocalDays` のJST化に対し、UTC/JST境界（深夜・月初/月末）でずれないテストを先に固定（DoD-06、[[予約モバイル仕様 AC-9]] と整合）。
- **認可回帰（必須）**: `role` に触れる表示分岐のため、マネージャー閲覧専用（入力導線非表示）の回帰テストを追加（CLAUDE.md セキュリティ不変条件）。
- **料金ロジック非変更の担保**: 明細の金額・負担割合導出が変わらないことのスナップショット/単体テスト。
- **E2E(Playwright)**: モバイルviewport（375×812）で「サマリー把握→売上推移カード確認」「入力フォームで明細追加→保存」。`src/__tests__/e2e-playwright/`、`PLAYWRIGHT_BASE_URL`・seed前提。
- 既存CI必須ゲート（`test:pr05:focused`）を壊さないこと。
- **壊れた実装に合わせてテストを変えない**（CLAUDE.md）。

## 10. 非機能・アクセシビリティ

- タップ領域44pt（`touch-target-comfortable` を標準化）。
- フォーム入力は `label` 関連付け・`aria-invalid`・エラーメッセージの関連付けを維持/付与。
- 本文16px以上・コントラストAA。数値入力は `inputmode` を適切化。
- パフォーマンス: 明細件数は1日の施術件数規模を想定。極端な大量時の仮想化は将来検討（§12-Q5）。

## 11. リスクと緩和

| リスク | 緩和策 |
|---|---|
| 料金/保険ロジックの巻き込み事故 | 表示層のみ変更。算出関数は触れず、非変更テストで担保 |
| 集計表示の数値差異 | API/RPCは非変更。カード化は表示の組み替えに限定 |
| SSR/hydration不整合 | `window.innerWidth` 直読みを避け、CSS優先 or `useMediaQuery` に一元化（[[responsive-table課題]]） |
| JST化による境界ズレの再混入 | 境界テストを先に固定してからリファクタ |
| マネージャー閲覧専用の崩れ | 分岐条件は変更せず、入力導線非表示の回帰テストを追加 |
