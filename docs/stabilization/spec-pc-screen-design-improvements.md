# spec: PC画面デザイン改善提案（非IT利用者向け直感操作）

- ステータス: 提案（未実装）
- モード: `EXTEND`（Design.md §1.0 — 既存トークン内での改善。REDESIGNではない）
- 対象: `(app)` 配下のPC表示（AppShell / サイドバー / ヘッダー / ダッシュボード / 予約 / 日報入力）
- 目的: **パソコンに詳しくない受付・施術スタッフが、迷わず・誤操作せず・毎日のルーチンを最短で終えられる**PC画面にする
- 根拠: Design.md パターンライブラリ（P01–P11）+ 行動心理学調査レポート（ヒック・ハイマン / フィッツ / 透明なデフォルト / 目標勾配 / ピーク・エンド）

## 方針

調査レポートの結論どおり、実装順は「**UI摩擦 → 意思決定 → 継続 → 記憶**」。まず P01（選択肢圧縮）/ P02（ターゲット拡大・近接）/ P03（透明な既定値）で土台を整え、その後 P05（進捗可視化）/ P11（完了体験）に進む。損失回避・希少性・アンカリング系（P08–P10）は院内業務システムには不要なので**使わない**。全提案は Ethics Gate（本人利益・可逆・事実性・拒否導線対等）を満たす低リスク施策のみ。

---

## Phase 1 — ナビゲーションの摩擦除去（優先度: 高 / 工数: 小）

### 1-1. サイドバーにアイコンを導入し、折りたたみ時の「2文字表示」を廃止

**現状の問題** (`src/components/navigation/sidebar.tsx`)

- メニュー項目の視覚手がかりが「●」の文字のみ（L147）。全項目が同じ見た目でスキャンできない
- 折りたたみ時は `item.label.slice(0, 2)`（L148）で「ダッ」「日報」「予約」「患者」のような先頭2文字表示になり、初見では判読不能
- 開閉トグルが「←」「→」の文字（L178）で、何が起きるか予測できない

**改善案**（P02 ターゲット / P01 認知負荷）

- `NavigationItem` に `icon`（lucide-react。既にプロジェクト採用済み）を追加: ダッシュボード=LayoutDashboard、日報=ClipboardList、予約=CalendarDays、患者=Users、収益=JapaneseYen、スタッフ=UserCog、シフト=CalendarClock、AI分析=Sparkles
- 展開時: アイコン+ラベル。折りたたみ時: アイコンのみ+`title`ツールチップ（2文字表示廃止）
- 開閉トグルは `ChevronsLeft` / `ChevronsRight` アイコン+aria-label
- アクティブ項目は背景色に加え左ボーダー（`border-l-4`）で現在地を明確化（既存 `primary-800` トークン内）

### 1-2. サブメニューの存在を可視化する

**現状の問題**: 「日報管理」「予約管理」にサブメニューがあるが、開くまで存在がわからない（chevron等の表示なし）。クリック時の挙動も「展開時=サブメニュー開閉、折りたたみ時=遷移」と不安定（L122-130）

**改善案**（P03 予測可能性）

- サブメニュー持ち項目に `ChevronDown` / `ChevronUp` を右端表示
- 現在ページを含む親メニューは初期状態で自動展開（現状は毎回手で開く必要がある）

### 1-3. ヘッダーの文字・絵文字ボタンをアイコンボタンへ統一

**現状の問題** (`src/components/navigation/header.tsx`)

- 「☰」（L345）「☀️/🌙」（L446）「▾」（L427）と文字絵文字が混在。通知・ユーザーはテキストのみでクリック可能に見えにくい
- ロゴ横に生のメールアドレスと英語ロール文字列（`clinic_admin` 等）を常時表示（L362-367）— 利用者には意味不明なジャーゴン

**改善案**（P02 / 一貫性）

- 通知=Bell、設定/管理=Settings、ユーザー=CircleUser、テーマ=Sun/Moon の lucide アイコン+ラベル併記（アイコンのみにしない: 非IT利用者はラベル必須）
- クリック領域は 44×44px 以上を担保（調査レポートの Apple HIG / Material 基準）
- ロール表示は `src/lib/constants/roles.ts` に日本語表示名マップを追加して「院長」「受付」「マネージャー」等で表示。メールはユーザーメニュー内へ移動

### 1-4. 店舗切替の「驚き」を除去

**現状の問題**: ヘッダーの店舗セレクタを変更すると、いきなり予約プレビューモーダルが開く（`header.tsx` L315-317 `setPreviewClinicId`）。「店舗を切り替えたら何か別の画面が出てきた」は非IT利用者が最も混乱する挙動

**改善案**（P03 透明な既定値・予測可能性）

- 切替時はモーダルではなく「◯◯院に切り替えました」のトースト表示に変更し、現在画面のデータを再取得
- プレビューが必要な管理者ユースケースは、トースト内の「予約状況を見る」ボタンに退避（opt-in化）

## Phase 2 — ダッシュボードの再構成（優先度: 高 / 工数: 中）

### 2-1. 「今日やること」を最上部へ（クイックアクションの位置と階層）

**現状の問題** (`src/app/(app)/dashboard/page.tsx`)

- クイックアクション（日報入力・予約確認・AIチャット）が**ページ最下部**（L283）。毎日使う導線に一番スクロールが必要
- 3ボタンが全て同色同格（L136-156）で、どれが主要か判断材料がない
- 異常値アラートも下部（L261）— 気づくべき情報が埋もれる

**改善案**（P02 近接 / P06 現在バイアス / P01 重み付け）

- 並び順を「①今日のあいさつ+日付+店舗名 → ②アラート（あれば）→ ③クイックアクション → ④本日データ → ⑤チャート類 → ⑥AIコメント」に変更
- クイックアクションは時間帯・状態に応じた1つを primary（塗り）、残りを outline に階層化。例: 日報未入力なら「日報入力」を primary
- AIチャットはPilot modeで無効化されるルート（`/chat`）なので、フラグ無効時は非表示にする（現状は押しても飛べないボタンが出る可能性）

### 2-2. PC幅を活かした2カラムレイアウト

**現状の問題**: シェルは `max-w-7xl`（`app-shell.tsx` L184）なのにダッシュボード本体は `max-w-4xl`（L242）で全カード縦一列。ワイド画面で左右が空白になり、スクロール量が増える

**改善案**（P02）

- `lg:` 以上で2カラムグリッド化: 左=本日データ+収益チャート、右=アラート+クイックアクション+AIコメント。混雑ヒートマップは全幅
- タイトルは「メインダッシュボード」→「おはようございます」+`toJSTDateString()` の日付+選択中店舗名（文脈の提示。`src/lib/jst.ts` 使用は必須規約）

### 2-3. ローディング・エラー体験

**現状の問題**: 読み込みはスピナー1個（L184-195）で画面全体が空白。エラー時は `window.location.reload()`（L224）の全リロードのみ

**改善案**（P05 進捗可視化 / P11）

- カード形状のスケルトン表示（レイアウトシフト防止+「何が出てくるか」の予告）
- エラーカードは React Query の refetch による部分再試行ボタンに変更

## Phase 3 — 業務フローの完了体験（優先度: 中 / 工数: 中〜大）

### 3-1. 日報入力のセクション化と進捗表示

**現状の問題**: `daily-reports/input/page.tsx` は1832行の単一ページで、施術明細・保険区分・支払方法など多数の入力が一画面に並ぶ。非IT利用者には「どこまでやれば終わりか」が見えない

**改善案**（P05 目標勾配 / P01 段階開示）

- 「①本日の明細確認 → ②金額・支払方法 → ③確認・保存」の3セクションに分け、上部に「残り◯項目」形式のステップ表示（調査レポート: 残量の具体化は「あと1分」より「残り2項目」）
- 予約データから自動生成された明細は「確認するだけ」の状態をデフォルトにし、手入力は追加操作に退避（P03: ユーザー利益が明確な透明デフォルト）

### 3-2. 保存後のピーク・エンド設計

**改善案**（P11）

- 日報保存・予約登録の完了時に「保存しました」だけでなく、①保存内容の要約（件数・合計）②次アクション（明日の予約を見る / ダッシュボードへ）③取り消し導線、をトーストまたは完了パネルで提示

### 3-3. ページヘッダーの共通化

**現状の問題**: 各画面がタイトル・戻る導線・アクションボタンを独自実装しており、位置と見た目が画面ごとに異なる（例: 予約画面は独自 `Header.tsx`、日報は `ArrowLeft` 手組み）。「どこにいるか」「どう戻るか」が画面ごとに変わるのは非IT利用者の学習コストを倍増させる

**改善案**（一貫性 / Design.md Section C）

- `PageHeader` 共通コンポーネント（タイトル / 説明 / 戻るリンク / 主要アクション枠）を `src/components/ui/` に追加し、主要画面から段階的に採用

## 用語の見直し（要ユーザー合意 — 実装前に確認）

サイドバーのラベルが分析ジャーゴン寄り（`src/lib/navigation/items.ts`）。非IT利用者向けには操作対象で命名する案:

| 現状 | 案 |
|---|---|
| 患者分析 | 患者 |
| 収益分析 | 売上 |
| スタッフ分析 | スタッフ |
| AI分析 | AIレポート |

※ 既存ユーザーの学習資産を壊すため、変更する場合はリリースノート告知とセットで。

## 使わない施策（Ethics Gate / 調査レポートの警告に基づく）

- 希少性表示・カウントダウン・損失回避コピー（院内業務システムに不適合、ダークパターン化リスク）
- preselection の拡大（保険区分・支払方法の既定値は「本人利益が明確・変更容易・表示明快」の3条件を満たす場合のみ）

## 計測（Design.md §Metrics）

- 主要: 日報入力完了までの所要時間 / ダッシュボード滞在からクイックアクション到達までのクリック数
- ガードレール: 誤操作起因の問い合わせ件数、予約の誤登録・取消率

## 実装順序とリスク

| Phase | 内容 | リスク | 影響範囲 |
|---|---|---|---|
| 1 | サイドバー・ヘッダー改善 | 低（既存トークン内、構造変更なし） | 全画面共通シェル |
| 2 | ダッシュボード再構成 | 低〜中（並び替え+グリッド） | /dashboard のみ |
| 3 | 日報ステップ化・完了体験・PageHeader | 中（フォームロジックに接触） | 日報・予約 |

各PhaseはDesign.md §1.2 の Design Rationale テンプレートを添えた個別PRとして実装する。
