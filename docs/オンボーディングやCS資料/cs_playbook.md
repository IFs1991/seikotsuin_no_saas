## 1. CSチームの役割と目標

### CSの存在意義

> 「プロダクトを売ること」ではなく、**「院が成果を出し続けること」**を支援する。

整骨院スタッフはITに不慣れな方が多い。プロダクトがいかに優れていても、「使えない」と感じた瞬間に解約が始まる。CSの仕事は、その「使えない」を潰し続けることである。

---

### CSチームが担う3つの責任

| 責任 | 具体的な行動 |
|------|------------|
| **成功の定義** | 各院が「何をもって成功とするか」を最初に合意する |
| **成功の実現** | タッチポイントを通じて、成功に向けた行動を促す |
| **成功の維持** | データで健全度を監視し、問題を早期発見・介入する |

---

### CSチームの主要KPI（詳細は Section 9）

- **NRR（Net Revenue Retention）**: 目標 110%以上
- **チャーン率（月次）**: 目標 2%以下
- **オンボーディング完了率**: 目標 85%以上（Day 30時点）
- **ヘルススコア 緑比率**: 目標 70%以上
- **NPS（Net Promoter Score）**: 目標 +30以上

## 2. 顧客セグメント定義

顧客をセグメントに分け、CSリソースの投下量を変える。

### セグメント区分

| セグメント | 定義 | CS対応モデル |
|-----------|------|------------|
| **Enterprise** | 3院以上（マルチ店舗利用） | ハイタッチ（専任CS担当） |
| **Growth** | 2院 または 月額プレミアム契約 | ミドルタッチ（月次定例あり） |
| **Starter** | 1院・スタンダード契約 | テックタッチ（自動化+必要時対応） |

---

### セグメント別の対応原則

#### Enterprise（ハイタッチ）
- 専任CS担当を1名アサイン
- 月次定例ミーティング（30分）を固定設定
- 四半期レビュー（QBR）を実施
- オンサイト訪問を年1回以上検討

#### Growth（ミドルタッチ）
- 月次メールレポートを自動送信
- ヘルススコアが黄色になったら担当から能動的に連絡
- 四半期に1回15分のビデオ通話を推奨

#### Starter（テックタッチ）
- オンボーディングはメール自動化シーケンスで対応
- ヘルススコアが赤になったときのみ担当から連絡
- インアプリのツールチップ・ポップアップでセルフサービス誘導

## 3. ヘルススコアリング（顧客健全度）

### スコアの考え方

ヘルススコアは「この院が今後も使い続けるか」を予測する指標。スコアが下がる前に介入することがCSの本質的な仕事。

---

### スコア構成（100点満点）

| 指標 | 重み | 測定方法 | 満点条件 |
|------|-----|---------|---------|
| **ログイン頻度** | 25点 | 過去30日のアクティブ日数 | 20日以上 |
| **日報入力率** | 20点 | 過去30日中の入力日数/営業日数 | 90%以上 |
| **予約データ入力** | 20点 | 過去30日に予約が1件以上入力された週数 | 4週中4週 |
| **機能活用度** | 20点 | 使用している主要機能数（ダッシュボード・予約・患者・日報・収益・スタッフ・AIチャットの7機能中） | 5機能以上 |
| **サポート問い合わせ頻度** | 15点 | 過去30日のクリティカル問い合わせ件数 | 0件 |

---

### ヘルスステータスの判定

| ステータス | スコア | 意味 | CSアクション |
|-----------|-------|------|------------|
| 🟢 **緑（健全）** | 75〜100点 | 定着・活用が進んでいる | 定期タッチ維持、拡張の提案 |
| 🟡 **黄（要観察）** | 50〜74点 | 使用が落ちてきている | 2週間以内に能動的コンタクト |
| 🔴 **赤（危険）** | 0〜49点 | 解約リスクが高い | 48時間以内に緊急介入 |

---

### ヘルススコア確認のルーティン

```
毎週月曜 → 全顧客のスコアを確認
  ├── 赤に変わった院 → 即日対応タスクを作成
  ├── 黄に変わった院 → 今週中に連絡タスクを作成
  └── 緑が3ヶ月継続 → アップセル検討リストに追加
```

## 4. オンボーディングプレイブック

### オンボーディングの成功定義

> **Day 30 までに「毎日開く習慣」が定着していること。**

具体的には以下の状態を目指す：
- ダッシュボードを週5日以上確認している
- 日報が週5日以上入力されている
- 予約データが入力されている
- 少なくとも3つ以上の機能を使っている

---

### Day 0：契約直後（24時間以内）

**CSがやること：**

- [ ] ウェルカムメールを送信（テンプレートA参照）
- [ ] 院長に最初のビデオ通話を設定（30分）
- [ ] セグメントを確認してCSチーム内でアサイン

**ウェルカムコール（30分）のアジェンダ：**

```
① 自己紹介・CSの役割説明（5分）
② 院の現状ヒアリング（10分）
   - 今どのようにデータ管理をしているか
   - 導入の目的・期待していること
   - 使い始めにあたっての不安・懸念
③ オンボーディングステップの説明（10分）
   - 手引書・FAQの案内
   - 30日ロードマップの共有
④ 最初の質問受付・次回タッチポイントの設定（5分）
```

---

### Week 1（Day 1〜7）：セットアップ完了

**目標**: 初期設定ウィザード完了 + スタッフ招待完了

**Day 3 チェックインメール（テンプレートB）：**

確認すること：
- ウィザードを完了できたか
- スタッフ全員がログインできているか
- 施術メニューの設定で詰まっていないか

**よくある詰まりポイントと介入：**

| 詰まりポイント | 確認方法 | 介入内容 |
|-------------|---------|---------|
| ウィザードが完了していない | ヘルススコア：ログイン0日 | 電話で画面共有サポートを提案 |
| スタッフが招待されていない | 招待ログを確認 | 招待手順をメールで再送 |
| 施術メニューが未設定 | マスターデータを確認 | 設定手順の動画/スクリーンショットを送付 |

---

### Week 2（Day 8〜14）：初回アクティブ確認

**目標**: 日報を5日以上入力、最初の予約をタイムラインで管理

**Day 10 チェックインメール（テンプレートC）：**

確認すること：
- 日報を入力できているか
- 予約管理をシステムで行っているか
- AIコメントが表示されているか（日報データが蓄積されるほど精度が向上する）

**Day 10 判定：**

```
日報入力5日以上 かつ 予約データあり
  → 「順調です！」メール送信 → Week3 プランの案内

日報入力0〜2日 または 予約データなし
  → 電話で状況確認 → 原因を特定して個別対応
```

---

### Week 3〜4（Day 15〜30）：活用定着

**目標**: 分析機能を使い始め、AIチャットで経営相談を体験

**Day 21 チェックイン（Enterprise/Growthのみビデオ通話）：**

確認すること：
- 患者分析を見ているか
- 収益ページを確認しているか
- 困っていることはないか

**Day 30 オンボーディング完了確認（全セグメント）：**

```
チェックリスト（Section 4の成功定義と照合）：
  [ ] ダッシュボード週5日以上のログイン確認
  [ ] 日報 20日以上の入力確認
  [ ] 3機能以上の活用確認
  [ ] NPS回答依頼メール送信
```

オンボーディング完了 → ヘルスステータスが🟢緑になっていれば成功。

---

### オンボーディングが失敗するパターン（要注意）

| パターン | シグナル | 対応 |
|---------|---------|------|
| **幽霊化** | Day 7時点でログイン0日 | 即日電話。「技術的な問題か」「操作方法がわからないのか」を丁寧に確認する |
| **院長だけが使っている** | 他スタッフのログインがゼロ | スタッフへの説明会セット（30分、オンライン可）を提案 |
| **データ入力しない** | 日報0件 | 「5分でできる日報入力デモ」動画を送付 |
| **設定で止まっている** | ウィザード未完了のまま | 画面共有で一緒に設定を完了させる |

## 5. 定期タッチポイント設計

### タッチポイント一覧

| タイミング | セグメント | 手段 | 目的 | 所要時間 |
|----------|-----------|-----|------|---------|
| Day 0 | 全員 | ビデオ通話 | ウェルカム・ヒアリング | 30分 |
| Day 3 | 全員 | メール | セットアップ確認 | — |
| Day 10 | 全員 | メール / 必要なら電話 | 初回アクティブ確認 | — |
| Day 21 | Enterprise/Growth | ビデオ通話 | 活用状況確認 | 15分 |
| Day 30 | 全員 | メール | オンボーディング完了確認・NPS | — |
| 月次 | Enterprise/Growth | ビデオ通話 | 月次レビュー・課題整理 | 30分 |
| 四半期 | Enterprise | ビデオ通話 | QBR（成果レビュー・目標再設定） | 60分 |
| 更新1ヶ月前 | 全員 | メール / 電話 | 更新確認・継続意向確認 | — |

---

### 月次レビュー（Enterprise/Growth）のアジェンダ

```
① 先月のハイライト（5分）
   - ヘルススコアの変化
   - 主要KPIの推移（売上・患者数・稼働率）

② 課題と解決策の確認（15分）
   - 先月のサポート問い合わせ内容の振り返り
   - 使いこなせていない機能の把握と改善提案

③ 来月の活用目標設定（5分）
   - 1〜2個の具体的なアクションを合意する

④ 新機能・アップデートの共有（5分）
   - リリースがあれば紹介
```

---

### 四半期レビュー（QBR）のアジェンダ

```
① 成果の振り返り（15分）
   - 導入前との比較（売上推移・患者数・業務時間）
   - 院として達成できたこと

② 課題と次の四半期の目標設定（20分）
   - 未活用機能の洗い出し
   - 次のフェーズで達成したいこと

③ プロダクトロードマップ共有（10分）
   - 今後の開発予定を共有

④ 契約・プランの確認（15分）
   - 現状のプランで満足しているか
   - 拡張の可能性（多院・追加機能等）
```

## 6. リスク顧客対応プレイブック（チャーン防止）

### チャーンの種類と対応の違い

| チャーンの種類 | 原因 | 対応の方向性 |
|-------------|------|------------|
| **プロダクトチャーン** | 機能不足・使いにくい | 機能要望を収集し、開発に連携。代替ワークフローを提案 |
| **価値未実感チャーン** | 成果を感じられていない | ROIを数字で可視化して見せる |
| **経営チャーン** | 院が閉院・経営悪化 | 情報を早期に掴む。プラン変更で残留を提案 |
| **競合チャーン** | 競合に乗り換える | 競合の優位点を把握。自社の差別化を再訴求 |

---

### 解約シグナルの早期発見

以下のシグナルが複数重なったら、ヘルススコアに関わらず介入を検討する。

```
⚠️ 解約シグナルチェックリスト
  [ ] ログイン頻度が先月比50%以上減少
  [ ] 日報入力が2週間以上止まっている
  [ ] サポート問い合わせが急増している
  [ ] 担当者から連絡への返信がなくなった
  [ ] 「検討します」「また連絡します」が増えた
  [ ] 請求に関する問い合わせが来た
```

---

### リスク顧客への介入プロセス

```
シグナル検知
    ↓
【48時間以内】CSから電話でコンタクト
    ↓
状況ヒアリング（以下を必ず確認）
  - 現在どのくらい使えているか
  - 使えていない理由は何か（時間・スキル・機能・価値）
  - 期待していたことに対して今どう感じているか
    ↓
原因別対応プラン作成（下表参照）
    ↓
フォローアップを1週間後に設定
    ↓
2週間後にヘルススコアを再確認
  → 改善 → 通常のタッチポイントへ戻す
  → 未改善 → 上位CSまたはマネージャーにエスカレーション
```

---

### 原因別対応プラン

| 原因 | 対応内容 |
|------|---------|
| 忙しくて使う時間がない | 「5分でできる日報入力」だけを継続してもらう。他は後回しでよいと伝える |
| スタッフが覚えられない | スタッフ向け説明会（30分オンライン）を提案 |
| 機能が足りない | 具体的な要望を聞いて開発チームに連携。代替方法があれば提案 |
| 価値を感じられない | 導入前後のデータを比較。数字で変化を可視化する |
| 他ツールとの使い分けが面倒 | 業務フローを整理して、このシステムの位置づけを明確にする |

---

### 解約申し出があった場合のプロセス

```
解約申し出の受付
    ↓
【当日】上位CSにエスカレーション（Section 8参照）
    ↓
解約理由を詳細にヒアリング
（この情報はプロダクト改善に必ず活用する）
    ↓
解約引き止め施策の提案（下記から選択）
  - プランのダウングレード提案（コスト削減）
  - 機能改善のロードマップ共有
  - 無料延長期間の提供（要上位承認）
  - 一時停止プランの提案（閉院・育休等の場合）
    ↓
引き止め成功 → 対応内容を記録し、フォロー強化モードへ
引き止め失敗 → オフボーディング手続きへ（データエクスポートを案内）
```

## 7. アップセル・拡張プレイブック

### アップセルのタイミング

アップセルは「関係ができた後」に行う。ヘルススコアが🟢緑で2ヶ月以上安定している院が対象。

---

### アップセル機会の見極め

| シグナル | 提案内容 |
|---------|---------|
| スタッフが5名以上になった | プラン変更（ユーザー数制限の拡張） |
| 「2院目を出したい」という発言 | マルチ店舗プランの提案 |
| CSVインポートを頻繁に要望する | データ管理オプションの案内 |
| AIチャットを毎日使っている | AI機能強化プランの案内（今後） |
| 「他のスタッフにも使わせたい」 | 追加ライセンスの提案 |

---

### アップセルの進め方

```
① 月次レビューやQBRの中で自然に話題にする
   「○○さんの院、2院目の計画があると以前おっしゃっていましたが…」

② 院が得ている価値を数字で確認してから話す
   「この3ヶ月で患者数が〇%増えていますね。このペースで行くと…」

③ 押し売りしない。「検討いただければ」で終わり、返答を待つ

④ 提案から2週間後にフォローアップ
```

## 8. エスカレーション手順

### エスカレーションの判断基準

以下のいずれかに該当する場合は、**その日のうちに**上位CSまたはマネージャーにエスカレーションする。

```
  [ ] 院からの解約申し出があった
  [ ] データ消失・セキュリティインシデントの可能性
  [ ] CSが3回コンタクトしても返答がない
  [ ] 院長から強い不満・クレームが来た
  [ ] 契約内容・請求に関するトラブル
  [ ] 法的な問題が示唆された
```

---

### エスカレーションの記録フォーマット

```
【エスカレーション報告】

院名：
担当CS：
発生日時：
エスカレーション理由：
  （上記チェックリストの該当項目を記載）
これまでの対応経緯：
現在の顧客の感情・温度感：
希望する対応・判断：
```

---

### エスカレーション先

| 状況 | エスカレーション先 |
|------|----------------|
| 解約申し出・強いクレーム | CSマネージャー |
| セキュリティ・データ問題 | 開発チームリード + CSマネージャー |
| 契約・請求トラブル | 営業・法務担当 |
| プロダクトへの重大バグ報告 | 開発チームリード |

## 9. KPI・成功指標

### CS全体のKPI

| KPI | 定義 | 目標値 | 計測頻度 |
|-----|------|-------|---------|
| **月次チャーン率** | 解約数 / 月初契約数 | 2%以下 | 毎月 |
| **NRR（純収益維持率）** | 既存顧客からの収益変化 | 110%以上 | 四半期 |
| **オンボーディング完了率** | Day30成功基準達成院の割合 | 85%以上 | 月次 |
| **ヘルスステータス緑比率** | 全契約院中の緑ステータス割合 | 70%以上 | 週次 |
| **NPS** | 推奨者比率 - 批判者比率 | +30以上 | 四半期 |
| **CSAT（問い合わせ対応満足度）** | サポート後アンケート | 4.0/5.0以上 | 毎月 |
| **TTR（問題解決平均時間）** | 問い合わせ受付から解決まで | 24時間以内 | 毎月 |

---

### CS個人のKPI

| KPI | 目標 |
|-----|------|
| 担当院のヘルス緑比率 | 70%以上 |
| 月次タッチポイント実施率 | Enterprise 100% / Growth 90% |
| 解約引き止め成功率 | 40%以上 |
| オンボーディング完了率（担当分） | 85%以上 |
| NPS回答収集率 | 60%以上 |

---

### 成果の可視化（院ごとのROIを数字で見せる）

院に「このシステムを使ってよかった」と感じてもらうために、定期的に以下の数字を院自身に見せる。

```
導入前（ヒアリングデータ）と比較する指標：

  ✅ 月次売上の推移（収益ページのデータ）
  ✅ 月間来院患者数の推移（日報データ）
  ✅ 離脱リスク高患者へのフォロー件数
  ✅ 予約管理にかかっていた時間（感覚値でOK）
  ✅ スタッフの稼働率の変化
```
